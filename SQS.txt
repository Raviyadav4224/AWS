1. What is Amazon Simple Queue Service (SQS) -
  
  * It is a fully managed Message Queue that lets different parts of your system to communicate
    Asynchronously 
    i.e It decouples the tightly coupled applications so that they can scale independently
    
2. Producers & Consumers -

  Producers (Senders) -

    * Any Service that sends messages to SQS Example - EC2, Lambda, Microservices etc

    * The message is produced to SQS using SendMessage API

    * The message is persisted in SQS until a consumer deletes it

    * Message retention - default 4 days, up to 14 days (Min - 1min, Max - 14 days, default - 4days)

  Consumers (Receivers) -

    * Service that reads and processes the messages stored inside a SQS is a consumer
      Example - EC2 workers, Lambda functions, Containers (ECS/EKS)
    
    * Consumers pulls messages So SQS is PULL based

    * Also SQS enables horizontal scaling, 1 EC2 - slow processing, Add 10 EC2 - process messages faster

    * Consumers uses ReceiveMessage API 
      It can receive up to 10 messages per request (batching) or Fewer messages

    * After processing, they must explicitly delete messages using DeleteMessage or 
      DeleteMessageBatch API

3. Types of SQS - 

  Standard Queue (Default) -

    * It provides Unlimited thoroughput, Low latency and unlimited number of messages in queue

    * Default retention of messages is 4 days, maximum 14 days

    * Limitation of 1024KB per message sent

    * Can have duplicate messages (Atleast Once Delivery)

    * Can have out of order messages (Best Effort Ordering)

  FIFO Queue -

    * It has STRICT Ordering

    * Exactly once processing i.e NO Duplication

    * It has lower thoroughput than Standard

4. SQS Security - 

  * Encryption -

    * In-Flight - Protects message while in transit from Producer to SQS and SQS to Consumer

    * At Rest - For data stored in SQS, uses AWS KMS for Encryption/Decryption 

    * Client Side Encryption - Customer manages the keys and performs Encryption/Decryption

  * IAM Policies - IAM policies to regulate access to SQS API's

  * Queue Access Policy -

    * It is similar to S3 bucket policies i.e Is this Queue allowing you?

    ** Used when you want to send message from Account A to Account B i.e cross account access
       or Allowing other services like Lambda, S3 etc to write to an SQS queue
  
  ** IAM vs Queue Access Policy -

    | Scenario                 | Use IAM  | Use Queue Policy   |
    | Same account user access | ✅       | ❌                |
    | Cross-account access     | ❌       | ✅                |
    | SNS → SQS                | ❌       | ✅                |
    | S3 → SQS                 | ❌       | ✅                |

  ** If service runs YOUR code - assumes IAM Role
     If service just sends events - needs resource policy

5. Message Visibility Timeout (Default 30sec) -

  * After a message is polled by a Consumer, it becomes INVISIBLE to other consumers
    By default the MESSAGE VISIBILITY TIMEOUT is 30sec i.e message has 30sec to be processed
    After the MESSAGE VISIBILITY TIMEOUT is over, message is visible in SQS again
  
  * If message is not processed within the Visibility timeout, it will be processed twice

  * Consumer could call ChangeMessageVisibility to get more time while message is still INVISIBLE
    If it is visible it cannot be called
    Example - 
      Receive message
      Start processing
      At ~25 sec - call ChangeMessageVisibility (extend by 60 sec)
      Finish work
      Delete message

  * If Visibility timeout is High, and the Consumer crashes, it will take hours to re-process the data
    On the other hand, if Visibility timeout is lower, we may get duplicates
  
  * Limit - Min 0sec, Max - 12 hours, Default - 30sec

6. Dead Letter Queue (DLQ) -

  * If a consumer fails to process a message within the Visibility Timeout, the message goes back to 
    the queue
    We can set a threshold of how many times a message can go back to the queue 
    After the MaximumReceives threshold is exceeded, the message goes into a Dead Letter Queue (DLQ)

  * DLQ of a FIFO queue must also be a FIFO queue
    DLQ of a Standard queue must also be a Standard queue

  * DLQ message retention is similar to SQS Queue i.e (Min - 1min, Max - 14 days, default - 4days)
    Example - If Message stays 2 days in main queue, Then moves to DLQ, It can still stay 14 more 
    days in DLQ

  * Redrive to Source - When our code is fixed, we can redrive the messages from DLQ back into the
    Source queue or any other queue without writing custom code

7. SQS Delay Queue -

  * It delays when a message becomes visible for the first time
    Example - Producer sends message, Message is stored in SQS, Message stays hidden for X seconds
    After delay - becomes visible - consumer can read

  * Delay can be configured at Queue Level or Message Level -

    * Queue-level delay -

      Applies to all messages in a queue
      Set using : DelaySeconds
      Range: 0 - 15 minutes (900 seconds), Deafult 0sec

    * Message-level delay -
      
      Override delay per message, by providing delay while sending the message
      Works per message
      ** Message level delay overrides Queue level delay

8. Polling in SQS (0 to 20sec) -

  * When a consumer requests messages from the queue, it can optionally wait for messages to arrive 
    if there are none in the queue, this is called Long Polling

  * LongPolling decreases the number of API calls made to SQS while increasing the efficiency and 
    decreasing the latency of your application.

  * The wait time can be between 1 sec to 20 sec (20 sec preferable)

  * Long Polling is preferable to Short Polling

  * Long polling can be enabled at the queue level or at the API level using ReceiveMessageWaitTimeSeconds
    Consumer

9. SQS Extended Client -

  * It is a Library that lets you send larger size messages via SQS by storing them in S3

  * Message size limit is 1024KB, in order to send larger size message upto 2GB, we need to use Extended
    Client Library
  
  * Producer sends large message
    Extended client uploads it to S3
    Sends a small SQS message with S3 reference
    Consumer reads SQS
    Fetches actual data from S3

  * Range - 256KB to 2GB

10. SQS API's -

  * CreateQueue (MessageRetentionPeriod), DeleteQueue
  * PurgeQueue: delete all the messages in queue
  * SendMessage (DelaySeconds), ReceiveMessage, DeleteMessage
  * MaxNumberOfMessages: default 1, max 10 (for ReceiveMessage API)
  * ReceiveMessageWaitTimeSeconds: Long Polling (How long the consumer waits to receive the message)
  * ChangeMessageVisibility: change the message timeout (0sec - 12hours)
  * Batch APIs helps decrease your costs

11. FIFO Queue -

  * It processes the messages in STRICT Order using Message Group ID, i.e messages in same Group ID are 
    processed in order
    Example - Message A,C with group-id 1,  Message B,D with group-id 2
    Group order-1 - processed in order (A - C)
    Group order-2 - processed in order (B - D)

  * Each Group ID can have different consumer thus enabling parallel processing
  
  * Message is processed Exactly once using Deduplication ID i.e same message send twice is - only one is
    accepted

  * We have Deduplication window of 5min, within this time, duplicate messages are ignored
    It is done in two ways -
      1. Using Deduplication ID
      2. Using Content based hashing - SHA256 of the message body

12. Amazon Simple Notification Service (SNS) -

  * It is used to send one message to many receivers 

  * It is based on Pub-Sub Model, and it is Push based 
    i.e Publisher sends message to SNS topic
    SNS pushes message to all subscribers
    Each subscriber processes independently

  * Publisher => SNS Topic => Multiple Subscribers, Publisher doesn't knows who the subscribers are

  * Topic - Logical Event, Example - User-signsup, order-events
  
  * Each subscriber to the topic will receives all the messages (Message filter can be applied)

13. SNS Security - 

  * Similar to SQS, i.e In-Flight Encryption for https requests, then IAM Policies for SNS APIs,
    At-rest Encryption and Client managed Encryption
  
  * SNS Access Policies (Similar to S3 bucket Policy) -

    Used for cross account access to SNS topics and by other Services to write to SNS

14. SNS + SQS = Fan Out Pattern -

  * Fan Out means 1 to N i.e same message is delivered to Many receivers, each having their own independent
    copies
  
  * Example -
    Event: OrderPlaced(orderId = 101)
    SNS topic: order-events
    Subscribers:
      Amazon SQS - Payment Queue
      Amazon SQS - Shipping Queue
      Amazon SQS - Analytics Queue
    SNS sends:
      - 1 copy to Payment Queue
      - 1 copy to Shipping Queue
      - 1 copy to Analytics Queue

  * It makes the services completely independent and No data loss, we can add more subscribers as well

  * Make sure to configure SQS Queue Access Policy for SNS to write

15. SNS FIFO -

  * Similar features like that of SQS FIFO i.e Strict ordering and No duplication using Message Group ID 
    and Deduplication ID
  
  ** We can have Standard SQS and FIFO SQS as subscribers to SNS FIFO topic

  * Example - OrderCreated event => SNS FIFO topic => Fan-out to multiple SQS FIFO queues (Payment, 
    Shipping, Billing) => each processes the event in order per orderId

16. Message Filtering in SNS -

  * JSON policy used to filter messages sent to SNS topic’s subscriptions

  * If a subscription doesn’t have a filter policy, it receives every message

  * Example -

    Buying Service (Order: 1036, Product: Pencil, Qty: 4, State: Placed)

    SQS Queue (Placed orders) , State: Placed Filter Policy
    
    Email Subscripxon (Cancelled orders), State: Declined Filter Policy
    
    SQS Queue (Declined orders), State: Cancelled Filter Policy

17. Kinesis -

  * Kinesis =  It is Real Time Data Streaming platform 
    i.e it continously Collects data (say logs, user events etc), processes it as it arrives(real time)
    and Pushes it to Destinations (like S3, Lambda etc)
  
    Kinesis has 3 main parts - Data Streams, Firehose and Data Analytics

18. Kinesis Data Streams -

  * Collects Streaming data and stores it temporarily (0 to 365 days, Default 24hrs), Multiple Consumers
    can read it
  
  * Shards = Capacity unit (Just like Pipe, one Pipe has some Capacity, to scale, we add more shards)
    i.e 1 shard = 1 MB/s WRITE and 2MB/s READ

  * Partition keys decides which shard the Data goes to
    Same key => same shard => Order maintained

19. Kinesis Data Firehose or Amazon Firehose -

  * It is a fully managed service of AWS

  * It takes the streaming data and automatically delivers to the Destination (like S3, Lambda, Redshift)

  * It has the ability to Re-process the data and data cannot be deleted until it expires

20. Kinesis Data Analytics -

  * It is serverless, fully managed service to process and transform streaming data in Realtime using
    SQL, Java, Python or Apache Flink etc

21. Kinesis Data Streams vs Kafka -

  * Core idea  (same in both) -
      
    Producers send events
    Data is stored in partitions/shards
    Consumers read data
    Multiple consumers can read same data
    Order is maintained within partition/shard
    
    | Kafka          | Kinesis         |
    | -------------- | --------------- |
    | Topic          | Stream          |
    | Partition      | Shard           |
    | Consumer Group | Consumer app    |
    | Offset         | Sequence number |

  * Difference -

    Kafka -

      You manage brokers, scaling, failures
      Add partitions (manual but flexible)
      Very high throughput possible

    Kinesis -

      Fully managed by AWS, No servers
      Add shards Each shard has limits: 1 MB/sec write, 2 MB/sec read
