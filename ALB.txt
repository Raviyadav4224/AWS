1. Scalability & High Availability - 

    Scalability -

        * There are two types of scaling

            Vertical Scaling -

                Increasing the capacity of instance  

            Horizontal Scaling -

                Adding/Removing more number of instances running the same application
                (= SCALE OUT (Add) / SCALE IN (Remove))

    High Availability -

        * High Availability goes hand in hand with Horizontal scaling

        ** High availability means running your application in atleast 2 Data centers or AZ's

2. Load Balancing with EC2 -

    * Load balancers distributes traffic across EC2's

    * It works hand in hand with ASG - ASG adds new EC2's and ELB routes traffic
    
3. Types of Load balancer -

    * Application Load Balancer -

        * Works on LAYER 7 (Application Layer) Http/Https

        * Supports Routing based on - Path, Host, Query strings, Headers

        * Supports HTTP, HTTPS, Web Sockets, HTTP/2, gRPC and redirect from http to https as well

        * Supports Load balancing to multiple applications across same and different machines
       
        * Has a port mapping feature to redirect to a dynamic port in ECS
        
        * ALB Target groups -

            EC2 instances (can be managed by an Auto Scaling Group) – HTTP 

            ECS tasks (managed by ECS itself) – HTTP 

            Lambda functions – HTTP request is translated into a JSON event

            IP Addresses – must be private IPs ( sending traffic to on premise instance inside VPC)

            ALB can route to multiple target groups

            Health checks are at the target group level

        ** Cross Zone Load Balancing is ENABLED by DEFAULT, can be disabled at Target group level
           No charges for inter AZ data transfer
           
    * Network Load Balancer -

        * Works on LAYER 4 (Transport Layer) TCP/UDP

        * Used for ultra low latency, non Http traffic 

        * NLB has one static IP per AZ, and supports assigning Elastic IP
          (helpful for whitelisting specific IP)
        
        * NLB are used for extreme performance, TCP or UDP traffic

        * NLB Target Groups - 

            EC2 instances
            IP Addresses - must be private IPs ( OnPremise instances inside VPC)
            ALB

        * Health check supports TCP, Http and Https protocols

        ** Cross Zone Load Balancing is DISABLED by DEFAULT
           You pay charges for inter AZ data transfer if enabled

    * Gateway Load balancer -

        * It works are Layer 3 (Network Layer) - IP Packets i.e Routing is done based on IP Addresses

        * It is used for managing fleet of 3rd party Network virtual applications in AWS

        ** It uses the GENEVA protocol on port 6081

        * GLB Target grpups -

            EC2 instances
            IP Addresses - must be private IPs

        ** Cross Zone Load Balancing is DISABLED by DEFAULT
           You pay charges for inter AZ data transfer if enabled

    * Classic Load balancer -

        * Supports Layer 4 (TCP) and Layer 7 (Http)

        * Health checks are TCP and Https based

        * Legacy, avoid in exams

        ** Cross Zone Load Balancing is DISABLED by DEFAULT
           No charges for inter AZ data transfer if enabled

    * Target Groups -

        Load balancers never talk to EC2 directly rather they talk to target groups

        Target groups can be EC2, Lambda(ALB only), IP Addresses
    
    * ELB continuously checks: /health, /status

4. SSL Certificates in Load Balancer & SNI -

    * Load balancer uses X.509 certificate

    * Without Server name indication ie SNI

        api.myapp.com - Requires one Certificate
        admin.myapp.com - Requires one Certificate

        Old behavior - One Load balancer = one Certificate
        i.e 1LB for api, 1LB for admin
    
    * With SNI, client sends hostname during TLS handshake, then LB chooses the cert based on hostname 
      or returns with default cert and routes traffic to correct target group
    
    * Classic Load Balancer (CLB) - Only 1 SSL certificate, No SNI support, Need multiple LBs for
      multiple certificates

    * Application Load Balancer (ALB) - Supports multiple certificates, Uses SNI

    * Network Load Balancer (NLB) - Also supports multiple certificates + SNI

5. Connection Draining or Deregistering Delay -

    * It means not breaking active user requests when instances are being removed

    * Say while user request is in process and AWS removes that instance, it will lead to timeout 5.XX
      errors so instances marked for removal are not instantly removed
      Rather no new request is send to the instance marked for removal and time is given for existing 
      request to complete
    
    * In CLB it is termed as Connection Draining
      In ALB it is termed as Deregistering Delay
    
    ** Time Delay ranges from 1 to 3600 seconds, default 300 seconds, can be DISABLED, set value 0

6. Auto Scaling Groups -

    * ASG's is a smart EC2 manager which adds/removes EC2 instance as and when required

    * ASG spreads instances across multiple AZ's so that failures can be reduced but NOT MULTIPLE REGIONS
      and ASG distributes instances evenly between multiple AZ's

    * ASG's Launch Template main components -

        AMI + Instance Type
        EC2 User Data
        EBS Volumes
        Security Groups
        SSH Key Pairs
        IAM Role for EC2 instances
        Network + Subnet information
        Load Balancer information

    * ASG supports EC2 health checks and ELB health checks

    * Desired/ Min/ Max capacity -

        Desired - target number of instances
        Min - lowest allowed
        Max - highest allowed

7. ASG Scaling Policies -

    * Dynamic Scaling Policy -

        * Target Scaling - 
            
            I want to keep the average CPU utilization to stay around 60%

        * Step Scaling - 

            When a Cloudwatch alarm is triggered (CPU utilization > 70%) - Add 2 Instances
            When a Cloudwatch alarm is triggered (CPU utilization < 30%) - Remove 1 Instance

    * Scheduled Scaling Policy -

        Anticipate scaling based on known usage patterns
        Say increase minimum capacity to 10 instances every Friday at 5pm

    * Predictive Scaling Policy -

        Scaling based on Load forecast, continuously forecast load and based on that scale out/in instances
    
    * Scaling parameters to consider - CPU Utilization, Average Network In / Out, Request count per target,
      Custom metrics

8. ASG Cooldown period -

    * After a scaling activity happens you are in cooldown period i.e no new instances are added or removed
      so that metrics could be stabilized
    
    ** DEFAULT cooldown period = 300seconds

    * Use ready to use AMI to reduce the cooldown time 

9. ASG Instance Refresh -

    * When you launch your new AMI with new configs, your existing EC2 instances dont automatically update
      So we need a way to replace the old instances, without downtime and in a controlled manner that is 
      what Instance Refresh does i.e it provides ways to replace or update older instances to new AMI in
      a controlled manner
    
    * When we trigger StartInstanceRefresh, ASG starts gradually replacing instances rather than all
      at once
    
    * We define warm up time i.e How long ASG waits before considering the new instance to be Ready

    * Minimum Healthy Percentage - 
        Example: ASG size = 10, Min healthy = 60% - At least 6 instances must always be running & healthy
        So ASG can replace only 4 at a time max and prevents downtime
