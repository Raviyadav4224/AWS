1. Relational Database Service (RDS) -

    * AWS RDS is a fully MANAGED relational database service provided by AWS

    * It allows us to create databases in -

        Postgres
        MySQL
        MariaDB
        Oracle
        Microsoft SQL Server
        IBM DB2
        Aurora 
    
    * AWS RDS advantages over DB deployed on EC2 - 

        Automated provisioning
        OS Patching
        Backup and Restore to specific timestamp
        Scaling capability (Vertical and Horizontal)
        Multi AZ support for Disaster Recovery
        Storage backed by EBS

    * AWS RDS is SINGLE AZ by default i.e One DB in one AZ, No Automatic Failover 
      Multi AZ needs to be enabled explicitly

    * You cannot SSH into your RDS DB instances except on RDS Custom

2. RDS Storage Auto Scaling -

    * It automatically increases the storage capacity of your RDS DB instances when it is running out of 
      free database storage
    
    * It Automatically modify storage if:
        Free storage is less than 10% of allocated storage
        Low-storage lasts at least 5 minutes
        6 hours have passed since last modification
    
    * You have to set Maximum storage threshold i.e Maximum storage limit for DB, If DB reached max
      storage limit it wont scale beyond that limit 

3. Read Replicas of RDS -

    * It allows us to create upto 15 Read replicas of our RDS DB 

    * It can be in Same AZ, Multi AZ or Cross Region as well

    * Replication is ASYNCHRONOUS, so reads are eventually consistent i.e with some lags 

    * Read replicas can be promoted to act as Actual DB i.e it becomes completely independent DB
      After DB promotion, there is no sync between old and new DB, data diverges i.e you have two
      different databases
    
    * In AWS, you pay when data goes from one AZ to another 

    ** Only the replication traffic that happens between the Primary DB and Read Replica within same Region 
       is free 
       ie Data copied from Primary RDS - Read Replica is FREE
       Even if: Different AZ (ap-south-1a → ap-south-1b) 
       Condition: Must be within the same region

4. RDS Multi AZ -

    * Muti-AZ = High Availability and NOT Scalability

    * We have - 
        
        One Primary DB (ACTIVE) - Handles all Reads/Writes
        One Standby DB (PASSIVE) - Exact copy of Primary DB, Kept in SYNC i.e Synchronous replication

        Primary in one AZ
        Secondary in different AZ
        Connected via Synchronous replication
    
    ** Unlike Read Replicas (ASYNC), in Multi AZ(SYNC) Write happens on primary, It is immediately copied 
       to standby, Only after both succeed response is sent
    
    * If primary crashes:

        AWS automatically detects failure
        Promotes standby - becomes new primary
        DNS endpoint remains SAME
        App reconnects - continues working
    
    **  You cannot read from standby
        You cannot access standby directly
        It’s not for scaling
        It’s purely a backup-in-real-time
    
    * Single AZ to Multi AZ - Just modify the database, Snapshot is taken from Single AZ DB, A new DB 
      is created using the Snapshot in new AZ and synchronization is established
    
5. Amazon Aurora -

    * It is a proprietary technology of AWS, Not an opensourced DB

    * Postgres & MySQL both are supported as Aurora DB, means Aurora will acts as if they were Postgres 
      or MySQL DB
    
    * It is 3X faster than Postgres and 5X faster than MySQL

    * It automatically increases storage in increments of 10GB upto 256TB

    * Failover in Aurora is instantaneous i.e it is Highly Available

    * It costs 20% higher than RDS

6. Aurora Architecture -

    * Aurora has two layers -

        Compute Layer (Your DB instance) -

            One Writer
            0-15 Readers
            Each instance runs in single AZ

        Storage Layer (Fully managed by AWS) -

            6 copies of Data across 3 AZ
            Always stores data in Multi AZ
    
    * Write & Read rule -

        Write - 4/6 - 4 copies out of 6 needed for writes
        Read - 3/6 - 3 copied out of 6 needed for reads

        * Even if one AZ fails - 4 copies writes works,>=3 copies read works
    
    ** If Aurora instance fails in one AZ, still the Data is not lost since it is stored in Multi AZ,
       i.e 6 copies of data is stored
       Just launch new Aurora instance and connect the same storage
   
7. RDS & Aurora Encryption -

    * By default data in RDS and Aurora is NOT encrypted, it must be enabled during launch time and encrytion 
      is done using AWS KMS
    
    * Why encrytion is not enabled by default - since data is generally huge in size it might lead to downtime
      if enabled by default
    
    ** If master is not encrypted, replicas CANNOT be encrypted - why - everytime replica will encrypt and 
       from master and will increase the complexity and performance
    
    * In Flight Encryption - By default both Aurora and RDS supports TLS/SSL encrytion but you must configure
      your client to use AWS TLS root certificate 

8. RDS Proxy -

    * It is fully managed database proxy for RDS

    * It sits between App and DB
      App - RDS Proxy - RDS / Aurora DB

    * DB has limited connections say 100-1000, Each connection consumes Memory and CPU, too many connections
      would lead to DB overload
      So RDS proxy maintains a pool of DB connections
      1000 users - Proxy - 50 DB connections

    * RDS proxy is never publically accessible, Always accessed through VPC

    * It supports both RDS and Aurora and reduces the failover time by 66% i.e supports automatic failover

    * When you should use this - 
        High traffic API's
        Serverless Apps i.e Lambda
        Connection spikes

9. ElastiCache -

    * It is an in-memory data layer to reduce the load on DB i.e it stores data in RAM and speeds up the 
      application
    
    * It supports two engines - Redis & Memcached
    
    * Use-cases - cache DB queries, pub/sub model, user session data, rate limiting etc

10. Caching Strategies in ElastiCache -

    * Lazy loading / Lazy population / Cache Aside -

        Application checks cache - if miss - call db - update the cache

    * Write-through -

        Write to Db and Cache together

    * Write-around -

        Write only to DB and later update cache 

11. Cluster Mode & Multi AZ -

    * By default disabled, Cluster is for SCALING

    * Cluster means redis node groups, 1 group contains 1 Primary instance + 0 - 5 Replicas

    * Cluster Mode enabled = Multiple redis node groups i.e more than one Primary instances

    * ElastiCache supports Multi AZ + Automatic failover and is used for High Availability
      If Primary fails replicas becomes the Primary

12. Amazon MemoryDB for Redis -

    * If node crashes, data can be lost, MemoryDB solves this problem by keeping Redis speed + makes data 
      durable like a database i.e Data in memory (fast), Also written to durable storage (distributed log)
      Multi-AZ by design
    
    * Scale seamlessly from 10s GBs to 100s TBs of storage

    * Use cases: web and mobile apps, online gaming, media streaming

13. SQL Endpoints in RDS & Aurora -

    * In RDS -

        Multi AZ - No need to change the connection string i.e same endpoint in case of failover
        Read Replicas - Separate endpoints, need to change manually

    * In Aurora -

        Multi AZ - No need to change the connection string i.e same endpoint in case of failover
        Read Replicas - Aurora gives two special endpoints 

            Cluster Endpoint - Always writes
            Reader Endpoint - automatically load balances across replicas

            Using one Reader endpoint, No need to manage replicas endpoint manually
    
14. Connecting to RDS / Aurora DB - 

    * There are three ways to connect to your database 
        
        1. Using Username and Password to directly connect to your DB

        2. IAM DB Authentication - Auth token based i.e IAM role is used to generate token and then 
           using Username and Token DB access is given 

        3. RDS Proxy - Instead of directly connecting to your DB using connection pooling using RDS proxy
           It also has same Username and Password way and IAM DB Authentication
