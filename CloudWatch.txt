1. AWS CloudWatch -

    * It is a Monitoring and Observability tool of AWS which helps to monitor the AWS services and
      provides Metrics, Logs, Events and Alarms
    

2. Core Pillars of CloudWatch (Metrics, Logs, Alarms & Events) -

    * Metrics -

        * It represents Numeric data of resources that measure the performance of the cloud services
          Example - CPU, Memory, Disk I/O etc
          EC2 => CPU Utilization, SQS => ApproxMessageVisibility
        
        * Types of Metrics -

            Default Metrics - 
                
                AWS gives you automatically Example EC2 CPU, Lambda invocations

            Custom Metrics - 
                
                Based on business requirement, we track our own metrics say no of orders
                Using put-metric-data we can push our custom metrics

        * Key terms -
            
            * Namespace - Logical boundary to group metrics Example AWS/EC2 - contains metrics related to EC2
                           
            * Metric - Variable to monitor

            * Dimensions - It is an attribute of a Metric, We can have up to 30 dimensions per metric
              Example - Metric CPUUtilization has dimensions like instance-id, Environment etc

            * Timestamps - Metrics have Timestamps
            
            * Metric Resolution - How fast the metric must be monitored and published/pushed

                Standard - 1min = 60sec
                High Resolution = Can have 1s / 15s / 30s - Higher cost

    * Logs -

        * Logs are detailed, timestamped records of events generated by the application or services
          Example - 
            App logs:
                User Ravi logged in
                Payment failed due to timeout

            Lambda logs:
                START RequestId...
                ERROR: NullPointerException

        * By default they are Encrypted, we can use AWS-KMS for client side encryption

        ** CloudWatch can Send logs to - S3, Kinesis Data Streams, Kinesis Data Firehose, Lambda,
           OpenSearch

        ** We can define log expiration policy from NEVER EXPIRE, 1 Day to 10 Years

        * CloudWatch Logs Sources - AWS SDK, CloudWatch Agent, CloudWatch Unified Agent, VPC Flow logs,
          ECS, Elastic beanstalk, Lambda, API Gateway, Route53 DNS Queries
        
        * Logs Insights - Search, Analyze and Query logs stored in CloudWatch Logs, it helps you to find
          specific IP, Error Count etc in the logs

        * CloudWatch Logs Export to S3 -

            * It is a BATCH process that copies log data from CloudWatch Logs into an S3 bucket for long
              term storage and analysis 
              Example - 
                Logs exist in CloudWatch Log Group
                You create an Export Task
                AWS copies logs => S3 bucket
                Logs stored as .gz files (compressed)

            ** It is not REAL TIME or NEAR-REAL TIME
               Log data can take up to 12 hours to become available for export
               It only exports Historical data, you must define start and end time
               S3 bucket policy must allow CloudWatch logs to write

        * CloudWatch Logs Subscriptions -

            Streams logs in Near Real time to downstream services for processing and analysis, send to 
            Kinesis Data Streams, Kinesis Data Firehose, Lambda
        
        * CloudWatch Subscription Filter -

            A subscription filter defines which log events are forwarded from a log group to a specified
            destination based on filter patterns

        * CloudWatch Logs Aggregation -

            CloudWatch Logs aggregation is an architecture pattern where logs from multiple accounts,
            services, or regions are centralized into a single destination (such as S3 or a central 
            logging account) for unified monitoring and analysis.

        * Cross Account Subscriptions -

            Cross-account subscription allows a CloudWatch Logs log group in one AWS account to stream log
            data to a destination in another AWS account.

        ** CloudWatch Logs Agent & Unified Agent -

            Logs Agent (Old & Deprecated) -

                It is python based log agent used only to send log files from EC2/On-Premise servers
                to CloudWatch Logs

                It only reads logs files and pushes to CloudWatch

            Unified Log Agent (Modern) -

                It is a single agent that collects both LOGS and SYSTEM LEVEL METRICS from EC2/On-Premise
                servers and send them to CloudWatch

                Logs like Application logs, System logs to AWS CloudWatch Logs

                Metrics like CPU, Memory, Disk usage, Network to AWS CloudWatch

    * Alarms -  

        * Alarms monitors a metric and performs an action when the threshold is breached
          
        ** It uses METRICS only and not Logs
        
        * Core Components -

            * Metrics - Like RAM, Memory, CPU etc
            * Threshold - Conditions like Errors>=5, CPU > 80%
            * Evaluation period - How long the condition must hold like CPU > 80% for 5 minutes
            * Action - Action to perform when threshold breaches
        
        * Alarm States -

            | State                 | Meaning            |
            | --------------------- | ------------------ |
            | OK                    | Everything normal  |
            | ALARM                 | Threshold breached |
            | INSUFFICIENT_DATA     | Not enough data    |

            Example - 
                CPU = 50% => OK
                CPU = 90% => ALARM
                No data => INSUFFICIENT_DATA

        * Types of Alarms -

            * Static Threshold Alarm - CPU > 80%
            * Anomaly Detection Alarm - Detects unusual behavior automatically, No fixed threshold
            * Composite Alarm - Combines multiple Alarms, CPU>80% and Error>5 => trigger
