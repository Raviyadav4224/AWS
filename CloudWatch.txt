1. AWS CloudWatch (Monitoring & Observability) -

    * It is a Monitoring and Observability tool of AWS which helps to monitor the AWS services and
      provides Metrics, Logs, Events and Alarms
    
2. Core Pillars of CloudWatch (Metrics, Logs, Alarms & Events) -

    * Metrics -

        * It represents Numeric data of resources that measure the performance of the cloud services
          Example - CPU, Memory, Disk I/O etc
          EC2 => CPU Utilization, SQS => ApproxMessageVisibility
        
        * Types of Metrics -

            Default Metrics - 
                
                AWS gives you automatically Example EC2 CPU, Lambda invocations

            Custom Metrics - 
                
                Based on business requirement, we track our own metrics say no of orders
                Using put-metric-data we can push our custom metrics

        * Key terms -
            
            * Namespace - Logical boundary to group metrics Example AWS/EC2 - contains metrics related to EC2
                           
            * Metric - Variable to monitor

            * Dimensions - It is an attribute of a Metric, We can have up to 30 dimensions per metric
              Example - Metric CPUUtilization has dimensions like instance-id, Environment etc

            * Timestamps - Metrics have Timestamps
            
            * Metric Resolution - How fast the metric must be monitored and published/pushed

                Standard - 1min = 60sec
                High Resolution = Can have 1s / 15s / 30s - Higher cost

    * Logs -

        * Logs are detailed, timestamped records of events generated by the application or services
          Example - 
            App logs:
                User Ravi logged in
                Payment failed due to timeout

            Lambda logs:
                START RequestId...
                ERROR: NullPointerException

        * By default they are Encrypted, we can use AWS-KMS for client side encryption

        ** CloudWatch can Send logs to - S3, Kinesis Data Streams, Kinesis Data Firehose, Lambda,
           OpenSearch

        ** We can define log expiration policy from NEVER EXPIRE, 1 Day to 10 Years

        * CloudWatch Logs Sources - AWS SDK, CloudWatch Agent, CloudWatch Unified Agent, VPC Flow logs,
          ECS, Elastic beanstalk, Lambda, API Gateway, Route53 DNS Queries
        
        * Logs Insights - Search, Analyze and Query logs stored in CloudWatch Logs, it helps you to find
          specific IP, Error Count etc in the logs

        * CloudWatch Logs Export to S3 -

            * It is a BATCH process that copies log data from CloudWatch Logs into an S3 bucket for long
              term storage and analysis 
              Example - 
                Logs exist in CloudWatch Log Group
                You create an Export Task
                AWS copies logs => S3 bucket
                Logs stored as .gz files (compressed)

            ** It is not REAL TIME or NEAR-REAL TIME
               Log data can take up to 12 hours to become available for export
               It only exports Historical data, you must define start and end time
               S3 bucket policy must allow CloudWatch logs to write

        * CloudWatch Logs Subscriptions -

            Streams logs in Near Real time to downstream services for processing and analysis, send to 
            Kinesis Data Streams, Kinesis Data Firehose, Lambda
        
        * CloudWatch Subscription Filter -

            A subscription filter defines which log events are forwarded from a log group to a specified
            destination based on filter patterns

        * CloudWatch Logs Aggregation/ Cross Region Observability -

            CloudWatch Logs aggregation is an architecture pattern where logs from multiple accounts,
            services, or regions are centralized into a single destination (such as S3 or a central 
            logging account) for unified monitoring and analysis
            i.e Cross Region Observability + Cross Region Dashboards

        * Cross Account Subscriptions -

            Cross-account subscription allows a CloudWatch Logs log group in one AWS account to stream log
            data to a destination in another AWS account.

        ** CloudWatch Logs Agent & Unified Agent -

            Logs Agent (Old & Deprecated) -

                It is python based log agent used only to send log files from EC2/On-Premise servers
                to CloudWatch Logs

                It only reads logs files and pushes to CloudWatch

            Unified Log Agent (Modern) -

                It is a single agent that collects both LOGS and SYSTEM LEVEL METRICS from EC2/On-Premise
                servers and send them to CloudWatch Logs

                Logs like Application logs, System logs to AWS CloudWatch Logs

                Metrics like CPU, Memory, Disk usage, Network to AWS CloudWatch

    * Alarms -  

        * Alarms monitors a metric and performs an action when the threshold is breached
          
        ** It uses METRICS only and not Logs
        
        * Core Components -

            * Metrics - Like RAM, Memory, CPU etc
            * Threshold - Conditions like Errors >= 5, CPU > 80%
            * Evaluation period - How long the condition must hold like CPU > 80% for 5 minutes
            * Action - Action to perform when threshold breaches
        
        * Alarm States -

            | State                 | Meaning            |
            | --------------------- | ------------------ |
            | OK                    | Everything normal  |
            | ALARM                 | Threshold breached |
            | INSUFFICIENT_DATA     | Not enough data    |

            Example - 
                CPU = 50% => OK
                CPU = 90% => ALARM
                No data => INSUFFICIENT_DATA

        * Types of Alarms -

            * Static Threshold Alarm - CPU > 80%
            * Anomaly Detection Alarm - Detects unusual behavior automatically, No fixed threshold
            * Composite Alarm - Combines multiple Alarms, CPU > 80% and Error > 5 => trigger

        * Period vs Evaluation Period -

            Period - 
            
                Period is how often the data is collected
                Example Period = 60sec, CloudWatch creates 1 data point after every 60sec

            Evaluation Period -

                How many data points are considered before triggering an alarm
                Example - Evaluation period 5, So CloudWatch would look at 5 data points
        
        * Data points to Alarms - 
            
            Means How many of those points must breach the threshold
            Example - Data point to alarm = 3 , Evaluation Period = 5, So if 3 data points breach
            out of 5, Alarm would be triggered
        
        * Alarms based on logs metrics -

            CloudWatch does not directly alarm on logs. We must convert logs into metrics
            Example - From logs extract pattern say ERROR : Payment Failed, Now based on this pattern 
            create a metric say ErrorCount, now based on this metric you can trigger an alarm

    * Events (EventBridge) -

        * Events = It helps you to React on something that happened in AWS
          Example - EC2 instance stopped, CodePipeline failed, S3 object uploaded
          These events are sent to: EventBridge (default bus)

        * Event happens => Default Event Bus => Rule matched => Action is triggered

        * EventBridge added major upgrades like -

            * Custom Application Events i.e Now it allows to send custom application events
                {
                    "source": "my.app",
                    "detail": {
                        "orderId": "123"
                    }
                }

            * Event Bus -

                Default Event Bus => AWS events
                Custom Event Bus => your app
                Partner Event Bus => SaaS (Shopify)
            
            * Schema Registry -

                Stores events structure(JSON), auto discovers from incoming Event

            ** We can send Cross Region / Cross Account Events to Event Bus, Resource policies must be 
               configured

        * Using EventBridge we can create a Rule with Schedule 

            Schedule Types - 
                
                rate() => Runs at fixed intervals
                cron() => specific time patterns (UTC by default)

4. AWS X-Ray (Troubleshooting) -

    * AWS X-Ray is distributed tracing of your application. It collects request data then lets you view,
      filter and inspect that data so that you can find errors, latency and slow downstream calls.

      For one traced request you can see not only the inbound request/response but also calls to AWS Resources,
      Micro services, Database calls and external web api's

    * It is a REGIONAL service i.e traces generated in one region stays in that region
      If your application is deployed across regions, traces are separated per region

    * It works with AWS Lambda, ECS, EC2, Elastic beanstalk, API Gateway, ELB

    * Core Concepts -

        * Trace - A complete end to end journey of one request across services

        * Segment - It is a JSON data recorded by ONE service for that request, Each service adds its own
          segment
          segments shows where time was spent and whether any errors occurred or not
        
        * Subsegment - Internal operations within the segment

        * Service Graph - Its a visual map which the X-Ray builds, It shows nodes, errors, latency etc

        ** Annotation vs Metadata - Annotations are simple Key-Value pairs that are indexed used for searching
           and filtering whereas Metadata are stored with trace but not indexed

          ** We can have upto 50 Annotations per trace
          We need to add Annotations manually in the code AWSXRay.getCurrentSegment().putAnnotation("userId", "UserA");
        
5. X-Ray Sampling -

    * By default AWS X-Ray doesn't records every request

    * It has default sampling rule - 
        Reservoir = 1 request per second (guaranteed, First request)
        Rate = 5% of additional requests
    
    * Example - 20 requests/sec => 1 guaranteed + 5% of 19 (~1) Total ~2 traces

    * We can define our own sampling rules based on business requirements

    * AWS says you can trace all requests for important WRITE operations and use a low sample rate for 
      high-volume READ-ONLY traffic like health checks or background polling to reduce COST

6. How to Enable X-Ray -    
    
    * Older Approach - Application (SDK) => X-Ray Daemon => AWS X-Ray Service

        1. Using SDK import the SDK -   

            * We need to instrument our application so that it can generate Segments/Subsegment, traces
              so that it can send to X-Ray
            
            * Example in Node.js -

                const AWSXRay = require('aws-xray-sdk');
                const express = require('express');

                const app = express();
                app.use(AWSXRay.express.openSegment('my-app'));

                app.get('/', (req, res) => {
                res.send("Hello");
                });

                app.use(AWSXRay.express.closeSegment());

        2. Installing / Enabling the X-Ray deamon to collect data from SDK -

        * SDK does NOT send data directly to X-Ray, It sends data to X-Ray Daemon, which then pushes
          to AWS
        
        * We can run daemon on 
          EC2 by installing manually, 
          ECS as sidecar or application container, 
          Elastic beanstalk just Enable X-Ray, 
          For Lambda just enable tracing, No daemon is needed
        
        ** Application (SDK) => X-Ray Daemon => AWS X-Ray Service
            Permissions (very important)
            Your compute must have IAM permission - xray:PutTraceSegments, xray:PutTelemetryRecords

    * Modern Approach - App (OpenTelemetry SDK) => ADOT Collector => X-Ray / CloudWatch / others
        
        * No X-Ray SDK needed
          No X-Ray daemon needed
          
          Uses:
            OpenTelemetry SDK
            OpenTelemetry Collector

    * Environment variables to communicate with X-Ray
      _X_AMZN_TRACE_ID: contains the tracing header
      AWS_XRAY_CONTEXT_MISSING: by default, LOG_ERROR
      AWS_XRAY_DAEMON_ADDRESS: the X-Ray Daemon IP_ADDRESS:PORT

7. AWS Distro for OpenTelemetry (ADOT) -

    * It is an AWS supported distribution of the OpenTelemetry and is vendor neutral

    * It can collect traces, metrics, and logs and can send it to multiple AWS Sources

    * It collects and transport i.e Application => OLTP => ADOT Collects/Transports => AWS X-Ray 
    
8. AWS CloudTrail (Inspect & Audit) -

    * It provides governance, compliance and audits of your aws account

    * We can get history of events or API calls made within your aws account

    * It is enabled by default

    * A trail can be applied to All regions(default) or single region

    * It can Audit API calls made by users / services / AWS console

    * It is useful to detect unauthorized calls or root cause of changes

    * Example - 
        
        Every action in AWS is an API call.
        Examples:
        Launch EC2
        Delete S3 bucket
        Create Lambda
        Update IAM policy
        Stop RDS
        Login to Console
        All these are API calls.

    * Event history - Stored for 90 days by default, for more time frame, store it in S3

    * Management Events vs Data Events -

        | Management Event      | Data Event                |
        | --------------------- | ------------------------- |
        | Create S3 bucket      | Upload object into bucket |
        | Create Lambda         | Invoke Lambda             |
        | Modify DynamoDB table | PutItem/GetItem           |
        | Lower volume          | High volume               |
        | Enabled by default    | Disabled by default       |

    ** It is REGIONAL but supports multi region by default
      Example - Consider IAM is global service, if someone in region attacks and CloudTrail is not enabled
      for that region than will miss the activity
