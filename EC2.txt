1. Elastic Cloud Compute (EC2) -

    * Rented virtual machines in AWS

    * EC2 = Amazon Machine Image (AMI) + Instance Type + Networking + Storage + Key Pair

    * EC2 always sits inside a Region/VPC i.e it is bound to AZ
     
        Region
            └── VPC
                └── Subnet (AZ-specific)
                    └── EC2 instance
    
    * Make EC2 highly available - ASG across multiple AZs

    * Whenever question says Need OS level access, custom software, legacy app - ans EC2

2. Amazon Machine Image (AMI) -

    * It is the template to launch EC2

    * It contains 
        
        OS (Amazon Linux, Ubuntu, Windows) 
        Preinstalled softwares
        Root volume
        Configurations
    
    * AMI are REGION specific, can be copied across regions, It can be AWS provided, Marketplace or
      custom
    
    * Launch identical EC2 instances quickly - create custom AMI's

3. Instance Types -

    * <ComputeFamily><Generation>.<Type>

        Examples - t3.micro, m5.large, c6g.xlarge
    
    * 
    | **Family**                        | **Prefix**             | **Optimized For**                | **Key Characteristics**       | **Typical Use Cases**               | **DVA-C02 Exam Signals**                                      |

4. Purchasing Options -

    * On-Demand Instances -

        Pay per second
        No Commitment 
        Most Flexible
        Highest Cost
        Used when - Short-term, Unpredictable workload

    * Reserved Instances -

        Standard RI - big discount, no flexibility

        Convertible RI - less discount, can change instance type

        Reserved for 1 or 3 Years

        Steady workload, predictable usage - Reserved Instance

        These instances provide discount upto 72% i.e 72% cheaper

    * Savings Plan -

        Commit $/hour, not instance type

        Applies to EC2, Lambda , Fargate

        More flexible compared to Reserved Instances

    * Spot Instances -

        Uses unused AWS capacity
        Can be terminated anytime with 2 minutes warning
        90% Cheaper
        Used for Batch jobs
        Fault Tolerant batch jobs - Spot Instance , Cheapest Compute

    * Dedicated Hosts vs Dedicated Instance -

        Dedicated Host: physical server (BYOL licensing)
        Dedicated Instance: isolated hardware, but AWS manages host
        Licensing question = Dedicated Host

5. Storage with EC2 -

    * Elastic Block Storage (EBS) -

        * It is AZ specific and NOT REGION specific i.e EBS created in one AZ, only EC2 instances in the
          same AZ can connect it
        
        * It is an Hard Disk attached to EC2

        * It persists data, AZ bound, can be dettached/reattached

        * Snapshots goes to S3

        * EBS Volume Types -

            | Type | Use case                    |
            | ---- | --------------------------- |
            | gp3  | Default, balanced           |
            | io2  | Mission-critical, high IOPS |
            | st1  | Throughput-heavy            |
            | sc1  | Cold data                   |
        
        * Root volume - deleted on termination (by default behaviour which can be changed)

            Data volume - usually preserved

        * EBS Volumes are not Encrypted by default, We need to enable it
        
        ** To encrypt exisiting unecrypted ebs volume we need to create snapshot of existing ones, 
           then recreate EBS using the snapshot and enabling encryption

        ** EBS volumes support both in-flight encryption and encryption at rest using KMS
        
    * Instance Store -

        Physically attached
        Ephemeral i.e for short time only
        Lost on stop/terminate
        Use only for: Temporary data, Caches
        Never for important data.

    * Elastic File Storage (EFS) -

        * Managed shared Network File System, works with Linux only, since Windows uses SMB protocol
          whereas Linux uses NFS protocol

        * Multiple instances can share the same EFS at the same time

        * Works across multiple AZ's

        * It is REGION Specific
            ├── AZ-A  → EC2 ──┐
            ├── AZ-B  → EC2 ──┼── EFS
            └── AZ-C  → EC2 ──┘

    * EBS Snapshots -
            
        * These are Point in time Backup of EBS volumes internally stored in S3, but not visible as 
          bucket objects
        
        * It stores Incremental Snapshots i.e First snapshot 100GB, next snapshot only changes = 2GB

        * It can be used to restore - Volumes, Create AMI, Copy accros regions

        * Backup EC2 data - EBS snapshot

        * Encrypted Volumes = Encrypted Snapshots
          Unencrypted Volumes = Unencrypted Snapshots or recreating another volume using snapshot 
          and enabling encryption creates encrypted volumes
          
6. Networking -

    * Elastic Network Interface (ENI) -

        Multiple ENIs
        Multiple private IPs
        Security groups attached to ENI
    
    * Public vs Private IP

        Private IP → inside VPC
        Public IP → changes on stop/start
        Elastic IP → static public IP
    
    * Security Groups -

        Stateful firewalls
        Allow only rules
        Evaluated First
        Instance level

        * Inbound - Default DENY
          Outbound - Default ALLOW

          Return traffic allowed - YES by default
    
    * NACL vs SG -

        NACL = stateless, subnet level

        SG = stateful, instance level

7. EC2 LifeCycle -  

    * Pending - Running - Stopping - Stopped - Terminated

        Stopped → EBS still billed
        Terminated → instance gone
        Instance store data lost on stop

8. Placement Groups -

    * Placement groups controls how EC2 are physically placed on AWS hardware inside one AZ

    * These are of three types -

        Cluster Placement Groups - Low latency, Single AZ
        Spread Placement Groups - max 7 instances per AZ, Multi AZ
        Partition Placement Groups - Instances split into logical partitions

9. Instance Meta Data Service (IMDS) -

    * AWS provides an internal service inside every EC2 instance called IMDS 
    
    * It is accessible at http://169.254.169.254/latest/meta-data

    * Metadata contains instance information like - Instance ID, Security groups, Private/Public IP,
      AMI ID, IAM role credentials, Availability Zone
    
    * We can also access User-data using http://169.254.169.254/latest/user-data

    ** In order to get instance details, we can use describe-api calls like
       aws ec2 describe-instances but it requires IAM role with required policies attached
       But it slower compared to instance meta data option

    * IMDSv2 requires Session Token to access the meta data
      TOKEN=$(curl -s -X PUT "http://169.254.169.254/latest/api/token" -H "X-aws-ec2-metadata-token-ttl-seconds: 21600")
      curl -s -H "X-aws-ec2-metadata-token: $TOKEN" -v http://169.254.169.254/latest/meta-data/
 
10. Key Points -

    * User Data Script - 

        Used while launching the EC2 instance
        It runs with root privileges
        Runs only during the boot cycle when you first launch the instance
    
    ** Environment variables vs IAM roles which credentials takes precedence -

        Environment variables > Shared Configs files (like ./aws/credentials, ./aws/config ) > IAM Role

        SDK uses environment variables
    
    ** Root Volume Termination -

        * By default root volume is set to DeleteOnTermination = true i.e when EC2 is terminated, root 
          volume also gets terminated

        ** We can change this setting, before launching the EC2 or When it is in Running state, using 
           CLI command with DeleteOnTermination = false
    
    * EC2 cannot directly assume an IAM role, AWS needs an intermediate object called Instance Profile
      1st Step => Create a Role => aws iam create-role
      2nd Step => Create Instance profile => aws iam create-instance-profile
      3rd Step => Attach Role to Instance profile => aws iam add-role-to-instance-profile
      4th Step => Attach Instance profile to EC2 => aws ec2 associate-iam-instance-profile

    *  The Maximum ratio of IOPS to volume size is 50 IOPS per GiB

    *   | Question says...               | Think...                                                  |
        | ------------------------------ | --------------------------------------------------------- |
        | 1 Reserved Instance            | One discounted instance at a time                         |
        | Multiple concurrent instances  | Extra ones are On-Demand                                  |
        | Runs only 1 minute             | Still eligible; billing is per second (60-second minimum) |
        | Compute Savings Plan           | Covers EC2 + Lambda + Fargate                             |
        | EC2 Instance Savings Plan      | Covers only eligible EC2 usage                            |
        | Unused Savings Plan commitment | You still pay the committed hourly amount                 |

    * For Linux EC2 instances -
      Minimum billing = 60 seconds (Both Reserved Instances & On-Demand Intances)
      After that = Per-second billing
    
    ** EC2 SSH Key Pairs -

        * Used to securely SSH into Linux EC2 instances

        * Key Pairs are Regional (not global)

        * When AWS creates a key pair, it generates:
            Public Key => Stored by AWS
            Private Key (.pem) => Downloaded by you only once

        * AWS never stores the private key for later retrieval

        * If the private key is lost, it cannot be downloaded again You must create a new key pair or 
          use another access method (such as Systems Manager Session Manager if configured)

        * To use the same SSH key pair in another Region, generate the public key (.pub) from your existing
          private key and import the public key into the other Region
          The same private key can then SSH into instances in all Regions where its matching public key 
          has been imported
        
    ** Regional Reserved Instance = Discount
       Zonal Reserved Instance = Discount + Capacity Reserved
       On-Demand Capacity Reservation = Capacity only (no discount)
    
    * EC2 monitoring -
      
      EC2 provides basic monitoring every 5min
      Detailed monitoring every 1min, needs to be enabled
    
    ** To enable monitoring -
        On running instances - aws ec2 monitor-instances --instance-ids
        On new instances - aws ec2 run-instances --montioring Enabled=true

    ** EC2 are AZ specific
       EBS are AZ specific
       EBS Snapshots are Regional
       AMI are Regional, to use in another region we can copy the AMI in new region
       ASG are Regional, can span multiple AZs
       Launch Templates are Regional, not global
       
       You cannot use a Launch Template directly in another Region, You must copy or recreate it and
       update any region-specific references

11. Multi-Volume Snapshots (EBS) -

    * It creates snapshots of multiple ebs volumes attached to an EC2 instance at one cooridnated time
      making the application consistent if the application is paused or suspended   
    
    * Need consistent backup of multiple EBS volumes attached to one EC2 - Go with Multi-Volume Snapshots

    * AWS gives you two ways to create snapshots -

      From Volumes => Snapshot one volume at a time
      From Instance => Snapshot all attached EBS volumes together (Multi-volume Snapshot)

    ** Multi Volume Snapshots creates one snapshot per EBS volume, but treats them as one coordinated
       backup operation
       Each snapshot is independent EBS snapshot and can be restored individually as well

12. EC2 Hibernation -

    * Hibernate - It freezes the machine exactly where it is instead of shutting down completely

    ** During Hibernation System Copies RAM to SSD (EBS) and Power off the CPU, later when system
       starts it reads the RAM from SSD and starts exactly where it left
    
    * RAM => copies => Root EBS Volume
      It literally stores RAM contents inside your EBS volume, Then CPU OFF, Memory OFF, Instance OFF
  
      Later Start Instance => Load RAM from EBS => Continue execution
      Programs never realize they stopped
    
    ** Charges during hibernation -

       Since machine is OFF, you are NOT Paying for RAM, CPU, Compute power
       You only pay for EBS since RAM image is stored there, Elastic IP if attached since AWS reserves that
       public IP for you

13. EFS vs EBS and its Security -

    * EFS - Multiple EC2 instances can connect and share data
    
    * EC2 connects to EFS over the network using Network File Sharing (NFS)

    ** Security Layers in EFS -

        * Security Groups - Network level protection, i.e who can access this

        * IAM Authorization - Using IAM policies, AWS decides Can this client mount? Can it write? 
          Can it read?

        * EFS File System Policies (Similar to S3 bucket policies) - Who can mount? Who can write? 
          Must use TLS? Allow Root?
        
        ** Suppose EC2 user root mounts EFS
           Without restrictions, root Can modify everything Dangerous
           AWS allows Deny ClientRootAccess inside the File System Policy
           Now even if someone is root on Linux, EFS says No, You are not root here
        
        ** File System Policies control read/write, That's only true when you're using IAM authorization
           (-o iam)
        
        * We can mount NFS in two ways -

            * Plain NFS (NO IAM) - 
                sudo mkdir -p /efs
                sudo mount -t nfs4 -o nfsvers=4.1 10.0.1.182:/ /efs
                EC2 => Security Group => NFS Protocol => POSIX Permissions
            
            * IAM Authorization -
                sudo mkdir -p /efs
                sudo mount -t efs -o tls,iam fs-092d1ffc8e064dd70:/ /efs
                EC2 => IAM Role => AWS Signature => EFS File System Policy => Security Groups => POSIX

    * Encrypted at Rest using KMS

    * EBS has no Security Group, The EC2's Security Group protects network traffic to the instance, 
      not the EBS volume

    * In EBS File access is controlled entirely by the operating system (POSIX permissions, ACLs, filesystem
      ownership etc)
