
1. Elastic Container Service (ECS) -

    * It is a fully managed Container Orchestration service offered by AWS to run, build, manage and 
      deploy docker containers
    
    * It handles -
        Running Containers
        Restarting Failed Ones
        Scaling
        Networking
        IAM Integrations
    
    * ECS doesn't store image, we generally use docker hub or ECR (Elastic Container Registry)

2. ECS Architecture -

    * It has 5 core components 

        Cluster
          Service 
                Task Definition
                    Container Definition
          Task
    
    * Cluster -

        It represents a logical group where containers runs
        It is just a grouping with no compute itself
    
    * Task Definition -

        * Task Definition = Blueprint i.e how should my container run

        * It defines 
            Docker Image
            CPU / Memory
            Ports
            ENV variables
            IAM Roles
            Logging configs
    
    * Task -

        * Task = Running Instance of a Task Definition
          example task definition = nginx server, task = running nginx container

    * Service -

        * It keeps the task alive i.e it ensures that X number of tasks are always running

        * Service ensures Load balancing & Auto Scaling

        * Example Desired count = 3, If 1 dies, ECS starts another
    
    * Launch Types - 

        1. EC2 Backed Launch Type -

            You manage the EC2 instances
            Cheaper at scale
            More control

        2. Fargate Backed Launch Type -

            You DON’T manage servers
            No EC2
            Pay per use
            Simpler
            ** It only supports AWSVPC network and doesn't works with bridge/host networks

    * ECS Task vs ECS Service -

        ECS Task -

            * A Task is the smallest unit of execution in ECS. It is instantiated from a Task Definition

            * Usage: Best for short-lived processes, such as data processing scripts or cron jobs, that have
              a clear start and end.

            * Persistence: If a standalone task stops or crashes, ECS will not attempt to restart it. 

        ECS Service -

            * An ECS Service is a configuration that allows you to run and maintain a specified number of task
              instances simultaneously in a cluster. 

            * Usage: Essential for production web servers or APIs that must be highly available.
            
            * Desired Count: You define a desired count, and the Service Scheduler ensures that if a task 
              fails, a new one is launched to replace it.

            * Traffic Management: Services can be configured with a Load Balancer (ALB/NLB) to automatically 
              route traffic to healthy tasks. 

3. Networking -

    * In ECS, Networking = How your container gets IP + how it communicates

    * Bridge Network -

        Containers run inside EC2
        Docker creates a bridge network
        Containers get private internal IP
        Traffic goes via port mapping
        Example Container runs on port 80, EC2 exposes 8080,Mapping: 8080 => 80
        User Access via: EC2-IP:8080

    * Host Network (only for EC2 launch type) -

        Container shares EC2 Network i.e container directly uses EC2 ports
        Example - Container uses port 80, EC2 also uses port 80

    * AWSVPC (default) -

        Each Task gets its own ENI & Private IP from VPC
        Example Task gets IP: 10.0.1.25, No port mapping needed
        Each Task in AWSVPC has its own Security Group
        It is default in ECS

4. IAM Roles in ECS -

    * Task Execution Role -

        * It is used by ECS itself and NOT your application code

        * It is used before the container starts

        * Example - Pull image from ECR, Send logs to CloudWatch, Authenticate with ECR etc
    
    * Task Role -

        * It is used by the application code i.e your application running inside the container

        * It is used after the container starts

        * Example - Read/Write to DynamoDb, Upload files to S3, Get Secrets from Secrets Manager 

5. Handson -

    * Phase 1: ECS with EC2 (Public Setup)

        User => ALB => Target Group (instance) => EC2 => ECS Agent => Task => Container
        Common Pitfalls
            Wrong health check path
            Port mismatch (80 vs 3000)
            SG misconfiguration
            App not binding to 0.0.0.0

    * Phase 2: ECS with Fargate (Public Setup)

        User => ALB => Target Group (ip) => Fargate Task (ENI) => Container
        | Feature      | EC2         | Fargate |
        | ------------ | ----------- | ------- |
        | Server mgmt  | You         | AWS     |
        | Target group | instance    | ip      |
        | Network mode | bridge/host | awsvpc  |
        | Scaling      | ASG         | managed |

    * Phase 3: IAM Roles (Real Usage)

        Project 1: S3 Access

            Create S3 bucket
            Upload / List files
            Create task role with:
            s3:GetObject or s3:ListObject
            Node app => fetch file using AWS SDK

        Project 2: Secrets Manager

            Store DB password in Secrets Manager
            Attach permission to Task role and Task Execution Role to add as env variables
            Fetch inside app as env variables

    * Phase 4: Networking (Public => Private)

        Internet => ALB (Public Subnet) => ECS Tasks (Private Subnet) 
        VPC Private Subnet must have NAT gateway or VPC Endpoints configured

6. Auto Scaling (Similar to ASG of EC2) -

    * ECS Service Auto Scaling handles TASK scaling, while EC2 ASG handles INFRASTRUCTURE scaling. 
    
    * In EC2 launch type both must be coordinated 
        Example => ECS scales tasks from 2 to 6, But ASG still has only 1 EC2 instance, Tasks go into 
        PENDING, No capacity to run them, whereas in Fargate only task-level scaling is required

        | Feature         | EC2 ASG        | ECS Auto Scaling    |
        | --------------- | -------------- | ------------------- |
        | Scales          | EC2 instances  | Tasks (containers)  |
        | Layer           | Infrastructure | Application         |
        | Used in Fargate | ❌ No           | ✅ Yes               |
        | Used in ECS EC2 | ✅ Yes          | ✅ Yes (both needed) |

7. Deployment Strategies -

    * Rolling Updates (Default in ECS) -

        * Replace old tasks gradually with new ones.

        * We need to define Minimum healthy percent, Maximum percent
            Example - Desired count = 2, Min healthy = 50% of Desired Count => 1 task must stay alive, 
            Max percent = 200% of Desired Count => up to 4 tasks allowed temporarily

        * Example 
            
            Start with - Old: 2 tasks

            ECS launches new tasks:
                Old: 2
                New: 1 => total = 3 (allowed, since max = 4)

            Once new task is healthy: ECS stops 1 old task

            Repeat:

            Old: 1
            New: 2

            Final:

            Old: 0
            New: 2

    * Blue/Green Deployment -

        * In ECS it is done through AWS CodeDeploy

        * Run two full environments -

            Blue = current live app
            Green = new version
        
        * Flow -

            Blue (current) is serving traffic
            Green (new version) is deployed separately
            Test green (internally or via test listener)
            Switch traffic instantly or gradually

        * Traffic switching options -
            
            Instant (all => green)
            Canary (10% => 50% => 100%)
            Linear shift

        * If something breaks - Rollback = switch traffic back to blue

    * Canary Deployment -

            * Release the new version to a small % of users first, then gradually increase.

            * Instead of replacing everything (rolling) or switching everything (blue-green),
              you test on real traffic in controlled chunks.
            
            * Canary is not just Deployment, it also includes Observability i.e You must monitor before
              increasing the traffic

            * Example -

                Desired tasks = 10
                New version deployed alongside old
                
                Step-by-step:
                Start:
                Old (v1): 100% traffic
                New (v2): 0%

                Shift small traffic:
                v1: 90%
                v2: 10%
                

                If stable:

                v1: 50%
                v2: 50%

                Final:
                v1: 0%
                v2: 100%

8. Data Volumes in ECS -

    * ECS volumes solve the same problems as Docker volumes plus infrastructure concerns
    
    * Storage depending on launch type -

        * EC2-backed tasks - 
        
            Amazon Elastic Block Store (attached to EC2)
            Data is tied to EC2 instance lifecycle
            If instance is terminated, data gone (unless separately managed)

        * Fargate-backed tasks -

            Ephemeral storage (default 20 GB, up to 200 GB)
            Data tied to task lifecycle
            Task stops => data gone

    * Bind Mount -
        
        A mapping mechanism, not a storage type

        Host path  =>  Container path

    * EFS -

        * Used to share common data between multiple containers across multiple tasks or Services

        * It must be configured explicitly and is not related to bind mounts

        * Data is persistent

9. Task Placements in ECS (Only for EC2 Launch Type) -

    * When ECS task is started with EC2 Launch type, ECS must determine where to place this task and the
      constraints of CPU and Memory(RAM)

      Similarly when a service scales in, ECS must decide which task to terminate

      For this we can define -

        Placemenet Strategy and Placemenet constraints
    
    * Task Placement Strategies -

        * Strategy = How to distribute tasks

        * Types of Strategies -

            * Binpack -

                Tasks are placed on the least available amount of CPU and Memory
                Minimizes the number of EC2 instances in use (Cost Saving)
                Example - "placementStrategy":[
                    { 
                        "type":"binpack",
                        "field":"memory"
                    }
                ]

            * Random -

                Tasks are placed randomly
                Example - "placementStrategy":[
                    { 
                        "type":"random",
                    }
                ]
                
            * Spread -

                Tasks are placed evenly based on the specified value like instanceId, attribute:ecs.availability-zone
                Example - "placementStrategy":[
                    { 
                        "type":"spread",
                        "field":"attribute:ecs.availability-zone"
                    }
                ]

            * We can combine multiple strategies as well
              Example - "placementStrategy":[
                     { 
                        "type":"spread",
                        "field":"attribute:ecs.availability-zone"
                     },
                     { 
                        "type":"binpack",
                        "field":"memory"
                     }
                ]
    
    * Task Placement Constraints -

        * Constraints = Where it is allowed to run

        * distinctInstance -

            Tasks are placed on a different EC2 instance
            Example - "placementConstraints":[
                    { 
                        "type":"distinctInstance",
                    }
                ]

        * memberOf -

            Tasks are placed on EC2 instances that satisfy a specified expression
            Uses the Cluster Query Language (advanced)
              Example - 
              
                "placementConstraints":[
                        { 
                            "type":"memberOf",
                            "expression":"attribute:ecs.instance-type =~ t2.*"
                        }
                    ]

                "placementConstraints":[
                    { 
                            "type":"memberOf",
                            "expression":"attribute:ecs.availability-zone in [ap-south-1a, ap-south-1b]"
                        }
                ]

    * Constraints = Hard Rule, If constraints fails = ECS doesn't places the task

      Strategies = Soft Rule, If strategies fails, task is still placed but not optimized

      If both are used -
        1. Constraints applied first
        2. Strategies applied second

13. Elastic Kubernetes Services (EKS) -

    * Amazon EKS = Amazon Elastic Kubernetes Service 

    * It is a way to launch managed Kubernetes clusters on AWS

    * Kubernetes is an open-source system for automatic deployment, scaling and management of containerized
      (usually Docker) application 

    * It’s an alternative to ECS, similar goal but different API

    * EKS supports EC2 if you want to deploy worker nodes or Fargate to deploy serverless containers

    * Kubernetes is cloud-agnostic i.e can be used in any cloud – Azure, GCP
