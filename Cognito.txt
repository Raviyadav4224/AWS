1. AWS Cognito -

  * Cognito = Authentication + Authorization for Applications

  * IAM is for AWS dev/admins/services whereas Cognito is for End Customer of your application

  * It consists of two main parts -

    Cognito User Pools - Authentication (Who are you?)
    Cognito User Identity - Authorization to AWS Resources (What are you allowed to access)

  ** Cognito Push Synchronization feature is used to Synchronize user data updates across multiple devices
     (Example user updates theme preference on mobile as dark, same must be reflected in Desktop)
     It doesnot interupts the user experience and works in background
     Although it is an older solution, Now a days AppSync is used for these purposes

  ** Cognito provides ADAPTIVE AUTHENTICATION i.e supports MFA, RisRisk-based sign-in blocking and Detects 
     compromised creds etc

2. Cognito User Pools -

  * It is a managed user directory which stores user related informations like username, email, MFA 
    settings, signin methods etc
  
  * It has various features such as signin/signup , password management, Social media login, MFA etc

  * After login Cognito gives JWT tokens (ID Token, Access Token & Refresh Token)

  ** Cognito can also provide ready made Login page i.e Hosted UI

  ** We can hook a Lambda function to auth life cycle i.e we can trigger lambda on pre-sign up to 
     authenticate user, post confirmation to send welcome email etc
  
  ** A user pool is required to manage users, authentication, and security policies in Cognito
     Without a user pool, you cannot enforce password resets or advanced security features

  * Cognito trusts external identity providers like Google, Facebook, OIDC, SAML, user pools
    
    For users directly signup using Cognito, Cognito stores user details + password
    
    For users from federated login's, Cognito stores user info like id name, but doesn't manages the 
    password

  ** Cognito Identity Pool can directly work without User Pool, Identity pool can directly federate with
     Google, Facebook etc

3. Cognito User Identity -

  * It provides temporary AWS credentials to Cognito Users instead of hardcoding the AWS keys

  * User logs in -> gets token
    Identity Pool validates token
    STS generates temporary AWS creds
    User accesses AWS resources

  ** Identity pools uses the IAM Roles 
    | User Type           | IAM Role    |
    | ------------------- | ----------- |
    | Authenticated users | Auth role   |
    | Guest users         | Unauth role |

  * It integrates well with API Gateway and Application Load balancer -

  * We can configure API Gateway => Cognito User pool authorizer
    No custom auth code is required
    API Gateway validates the Cognito JWT Token
  
  ** We can use Custom Domain instead of AWS hosted UI Domain
     ACM Certificate is to be created in us-east-1 for using custom domain

  ** Unauthenticated Users -
    
    * Guest Users needs access to AWS services without login or signup 

    * Browser directly accessing Dynamodb needs Cognito Identity without signup and not IAM Role for 
      getting temporary credentials
      IAM Role is for EC2 server access

    * This can be achieved using Cognito Identity pools for Unauthenticated users and NOT Cognito User pool

  * To assign unique identifiers to Users regardless of the device they use, We can Implement 
    developer-authenticated identities by using Amazon Cognito, and get credentials for these identities

4. AWS Security Token Service (STS) -

  * Suppose you have AWS account A
    Your frontend app needs S3 access
    Or EC2 instance needs DynamoDB access
    Or a user from another AWS account wants access
    Or a CI/CD pipeline needs temporary deployment access

    You should NOT -

      hardcode AWS access keys
      share permanent credentials
      create IAM users everywhere

    Instead -

      AWS gives temporary credentials
      valid for limited time
      automatically expire

  * AWS STS generates - Access Key ID, Secret Access Key, Session Token
    These are called Temporary Security Credentials
    They expire automatically after 15 mins, 1 hour, 12 hours, depending on config

  ** Important API's of STS -

     AssumeRole => Assume another IAM role (Generates temporary credentials)
     AssumeRoleWithWebIdentity => Google/Facebook/Cognito login (Generates temporary credentials)
     GetSessionToken => Temporary creds for IAM user with MFA
     GetCallerIdentity => It tells Who am I currently authenticated as? It returns accountId, ARN, user/role

  * EC2 IAM Roles, uses STS to get temporary credentials
    AWS Cognito uses AWS STS to get temporary credentials

  ** Using get-session-token api gives 
     {
      "Credentials": {
        "AccessKeyId": "ASIA....",
        "SecretAccessKey": "abc....",
        "SessionToken": "IQoJb3J....",
        "Expiration": "2026-05-22T18:00:00Z"
      }
    }

  * In order to access other AWS Resources - AccessKeyId, SecretAccessKey, SessionToken is mandatory, 
    using session token AWS knows that these credentials came from STS, session is valid and not expired

  * AWS STS decode-authorization-message API is used to decode the message which aws sends in encoded
    format

5. AWS KMS & Encryption -

  * Encryption at Rest -
    
    * Data is encrypted before storing on disk
  
  * Encryption in Transit -

    * Data is encrypted while it is moving over the network usually using HTTPS/TLS or SSL

  * Client Side Encryption -

    * Client encrypts the data and then sends to AWS and is done at Rest

  * AWS KMS -

    * It is a Key management service, which manages storing, accessing, controlling access, auditing of
      keys
    
    * These keys can be AWS generated or User generated

    * Types of Keys -

      * AWS Owned (SSE-S3) -

        AWS manages these keys internally 
        You do NOT control - lifecycle, rotation policy, permissions, key deletion
      
      * AWS Managed Keys (SSE-KMS) -

        Keys are created automatically in your account and AWS manages the keys
        You control - key policies, rotation, enable/disable, deletion
        
        * It also provides you with an AUDIT trail of when your key was used and by whom
      
      * Customer Managed Keys (SSE-C) -

        * User creates and manages them, has more control over rotation, iam control, cross account 
          access etc
        
        * AWS doesn't stores the Key, Data is encrypted at Rest

    * KMS API's can throttle at very high GET/PUT rates
      SSE-KMS every encrypt/decrypt calls KMS API's
    
    * When Uploading Objects to S3, Headers tell S3 which Encryption to use -

      SSE-S3 => x-amz-server-side-encryption: AES256
      
      SSE-KMS => 
        x-amz-server-side-encryption: aws:kms
        x-amz-server-side-encryption-aws-kms-key-id: <kms-key-id>
      
      SSE-C =>
        x-amz-server-side-encryption-customer-algorithm
        x-amz-server-side-encryption-customer-key
        x-amz-server-side-encryption-customer-key-MD5
  
    ** Using SSE-KMS as encryption and enabling Bucket key we can reduce AWS KMS Api calls for every object
       It only works for SSE-KMS and NOT SS3-S3, SSE-C(since KMS is not involved here)
    
    ** Data Key Caching feature is available with AWS Encryption SDK encryption library which reduces
       KMS API calls

    * KMS Throttles - GenerateDataKey => Data Key Caching (LocalCache) + AWS Support to increase Quota

  ** Encryption/Decryption Process or Envelope Encryption -

    For Large files (Reduces KMS API calls) -

      During Encryption -

        1. Request Data Key from KMS (Generate Data Key from KMS) Using Master Key
        2. KMS returns:
            - Plaintext Data Key
            - Encrypted Data Key
        3. Use plaintext data key locally to encrypt large file
        4. Delete plaintext key from memory
        5. Store:
            - Encrypted file
            - Encrypted data key
      
      During Decryption -
      
        1. Read encrypted data key
        2. Send encrypted key to KMS
        3. KMS decrypts data key
        4. Get plaintext data key temporarily
        5. Decrypt file locally

    * Master Key (encrypts) => Plaintext Data key + Encrypted Data key => Data encrypted by Data Key

  * AWS CloudHSM (Hardware Security Module) -

    * It is used when we need Maximum Control over the encryption keys
    
    * It is used generally in Banking, Govt, HealthCare applications
    
    * A dedicated hardware is provided for storing keys, supports lifecycle, backups, scaling, users

6. AWS Serverless Application Model (SAM) -

  * SAM = CloudFormation + Shortcuts for Serverless
    i.e using SAM we can create serverless application with lesser templates
  
  * It is built on the top of CloudFormation i.e it is an higher level abstraction of CloudFormation

  * All the configuration is YAML code

  * SAM natively uses CodeDeploy to update Lambda functions
    Using AutoPublishAlias, DeploymentPreference SAM auto configures Lambda function versions, aliases 
    and Rollout
    It provides Traffic shifting feature
  
  * API Gateway supports OpenAPI/Swagger API definitions for creation of API's
    The Swagger file can be referenced inside SAM template either inline or by defining a reference file

7. Components of SAM -

  * SAM Template = Infrastructure Definitions
  * SAM CLI = Build/test/deploy locally
  * Transform = Converts SAM templates to CloudFormation
  * Local emulation = Runs Lambda/API's locally

  * | Resource Type                   | Purpose                      |
    | ------------------------------- | ---------------------------- |
    |  AWS::Serverless::Function      | Lambda Function              |
    |  AWS::Serverless::Api           | API Gateway REST API         |
    |  AWS::Serverless::HttpApi       | API Gateway HTTP API         |
    |  AWS::Serverless::SimpleTable   | DynamoDB Table               |
    |  AWS::Serverless::StateMachine  | Step Functions State Machine |
    |  AWS::Serverless::LayerVersion  | Lambda Layer                 |
    |  AWS::Serverless::Application   | Nested SAM Application       |

8. SAM CLI Commands -

  | Command                | Purpose                                                          |
  | ---------------------- | ---------------------------------------------------------------- |
  | sam init               | Create project                                                   |
  | sam build              | Build dependencies (Creates .aws-sam/ folder with packaged app.) |
  | sam local invoke MyFn  | Invoke Lambda locally (Runs Lambda locally using Docker)         |
  | sam local start-api    | Run local API (Runs local API Gateway emulator)                  |
  | sam deploy             | Deploy app (Deploys CloudFormation stacks)                       |
  | sam package            | Upload artifacts to S3                                           |
  | sam sync               | Synchronize code and infrastructure (--code only code changes)   |

  ** sam deploy = includes sam package, packages artifacts, uploads to S3 and deploys CloudFormation stack
     sam package is legacy command now

  * Correct order of SAM commands to deploy - sam init, sam validate(optional), sam build, sam deploy

9. AWS Systems Manager Parameter Store vs AWS Secrets Manager -

  AWS Secrets Manager -

    It is for storing secrets like DB_Passwords and supports Automatic secret rotations, Versioning,
    staging labels
    It encrypts the data by default

  AWS Systems Manager Parameter Store -

    It is used to store application configs and doesnot supports secret rotation or versioning
    It is for simpler use cases and is cheaper compared to Secrets Store
    It can store either Strings or SecureStrings which are encrypted

10. AWS Certificates Manager -

  * AWS Certificate Manager (ACM) issues and manages SSL/TLS certificates
  
  * Used to enable HTTPS and verify whether the Domain actaully belongs to the Correct Owner or not

  * DNS Validation -

    ACM provides a CNAME Name and Value.
    Add the CNAME record to your DNS provider (Route53, GoDaddy, etc.)
    ACM validates domain ownership and issues the certificate
  
  * For CloudFront - Certificate MUST be created in us-east-1 if attaching to CloudFront

  * ACM Renewal -

    * ACM Issued Certificates - 

      * ACM Certificates are Automatically renewed, before expiration ACM requests new Certificate and 
        updates
      * ACM renews the certificate (60 days prior for public, 30 days for private) if associated with an
        integrated service
            
      * If the certificate is being used by an AWS service and ACM-provided CNAME records are accessible 
        via the public DNS, ACM will consider the domain name validated and auto renew the certificate

      * If the domain is not validated, it will send Health events or EventBridge events to notify the 
        domain owner prior to expiration


    * For Imported or 3rd Party Certificates - User must import the renewed certificate manually

11. Miscellaneous AWS Services -

  * AWS Inspector Service -

    It is a Deep security vulnerability scanning service
    Checks for  CVEs, vulnerable packages, insecure software versions
    Mainly for - EC2, Lambda dependencies, ECR container images

  * Trusted Advisor -

    It suggests AWS best-practices
    Suggests - cost optimizations, security improvements, fault tolerance improvements, 
    service limit warnings, performance recommendations
    It focusses on High-level AWS account hygiene and best practices

  * AWS Step Functions -
    Serverless workflow orchestration service used to coordinate multiple AWS services and Lambda functions
    Uses state machines and JSON-based workflows
    Handles retries, error handling, branching, parallel execution, and long-running workflows
    Exam keyword: "orchestrate multiple Lambda functions" → Step Functions.
    
  * AWS AppSync (GraphQL)
    Managed service for building GraphQL APIs
    Allows clients to request exactly the data they need from multiple data sources (DynamoDB, Lambda, RDS, etc.).
    Supports real-time updates via subscriptions and offline synchronization
    Exam keyword: "GraphQL API" → AppSync
  
  * AWS Amplify
    Full-stack development platform for building and deploying web/mobile applications
    Integrates easily with Cognito, AppSync, Lambda, S3, API Gateway, etc
    Provides hosting, CI/CD, authentication, storage, and APIs
    Exam keyword: "Frontend/mobile developer productivity" → Amplify

  * AWS AppConfig
    Feature of AWS Systems Manager used for dynamic configuration management
    Deploy application configuration changes without redeploying code
    Supports validation, gradual rollouts, monitoring, and automatic rollback
    Exam keyword: "Change config without redeployment" → AppConfig
    It polls whereas in Environment variables application restart is required to reflect variables changes
