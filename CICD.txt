1. AWS CodeCommit -

  * It is a fully managed version control system like GIT designed for developers, so that they
    push the code => CodeCommit stores the version => CI/CD uses the latest code => Deploy
    automatically
  
  * It provides private git repositories with No size limit and can scale seamlessly

  * It provides strong security like encryption and access control

2. AWS CodeCommit Security -

  * Authentication -

    1. HTTPS Git Credentials - AWS generates: Git username, Git password, Used with normal Git HTTPS
        URL

    2. SSH Authentication - Upload SSH public key to IAM user. Then use SSH clone URL
        git clone ssh://git-codecommit.ap-south-1.amazonaws.com/v1/repos/myrepo

    3. AWS CLI Credential Helper - Uses existing AWS CLI credentials automatically.
        git config --global credential.helper '!aws codecommit credential-helper $@'
        git config --global credential.UseHttpPath true

  * Authorization -   

    * IAM policies to manage users/roles permissions to repositories
    
    * User needs both Authorization + Authentication to access the CodeCommit

* Encryption -

  Repositories are automatically encrypted at rest using AWS KMS
  Encrypted in transit (can only use HTTPS or SSH – both secure)

  * Cross account access -

    Do NOT share your SSH keys or your AWS credentials
    Use an IAM Role in your AWS account and use AWS STS (AssumeRole API)

3. AWS CodePipeline -

  * It is a fully managed CI/CD Orchestration service

  * It consists of different stages like Source, Build, Test, Deploy, Invoke
    Each stage can have some actions and can run in parallel as well
    Example: Build => Test  => Deploy => Load Testing etc

  * Pipeline = sequence of stages

  * We can create Events using Amazon EventBridge for failed/cancelled pipelines

  * It also provides Manual Approval Action i.e pipeline waits for approval and after approval
    proceeds for the next step

  * Manual Approval supports SNS only 
    Pipeline reaches approval stage => CodePipeline publishes to SNS => Email/Lambda/SMS/HTTP
    subscribers notified

  ** CodePipeline cannot be configured as a trigger for Lambda
     CodePipeline can invoke a Lambda action on stage changes but Lambda cannot be triggered when an 
     event happens in CodePipeline i.e CodePipeline configured as Lambda Trigger Source is not possible
     
     Generally we follow CodePipeline State Changes => EventBridge => Lambda
  
  * CodePipeline provides many built-in actions (CodeBuild, CodeDeploy, Lambda, CloudFormation, etc.)
    If your requirement cannot be implemented using one of those built-in actions, you create a 
    Custom Action backed by a Job Worker which uses CodePipeline public endpoint to access the API's

4. AWS CodePipeline Artifacts -

  * It refers to the output files between stages
  * Each Pipeline stage can create Artifact like zipped build, test results etc
    These artifacts are stored in S3 Artifact bucket

5. AWS CodeBuild -

  * It is a fully managed continuous integration service (alternative to Jenkins) and is used to
    compile source code, run tests and create build
  
  * It uses buildspec.yml file or we insert manually in console for build instructions

  * Output logs can be stored in S3 or CloudWatch, We can use metrics to monitor build statistics

  ** To execute Docker commands inside CodeBuild we need to enable PRIVILEDGED MODE 
  
  ** By default CodeBuild runs inside AWS managed environment i.e Outside your VPC and thus -
     It cannot access your resources like RDS, EC2
     But it can access internet and public resources by default
     So we must allow VPC integration and is done at CodeBuild Project settings

  * To provide access we must provide VPC specific informations as a part of CodeBuild project
    configurations
    This includes the VPC ID, the VPC subnet IDs, and the VPC security group IDs
    VPC-enabled builds are then able to access resources inside your VPC
  
  * If CodeBuild is inside private subnet, it must have NAT Gateway to access internet or VPC endpoints 

  * CodeBuild artifacts can be configured to store in S3 bucket by specifying S3 artifact bucket
  
  ** Artifact created is ENCRYPTED by default using AWS SSE-S3(AES-256) which is default encryption of S3
     buckets
     To change encryption we can provide key using KMS
     Same applies to CodePipeline as well

6. AWS CodeBuild buildspec.yml file -

  * buildspec.yml file must be at the root of your code

  * We can define - 
    Environment variables
    Phases - install, pre_build, build, post_build 
    Caches and  artifacts

    Example -

      version: 0.2
      phases:

        install:
          commands:
            - echo running install phases
            
        pre_build:
          commands:
            - aws ecr get-login-password --region ap-south-1 | docker login --username AWS --password-stdin 781108903979.dkr.ecr.ap-south-1.amazonaws.com

        build:
          commands:
            - docker build -t node-api .
            - docker tag node-api:latest 781108903979.dkr.ecr.ap-south-1.amazonaws.com/node-api:latest

        post_build:
          commands:
            - docker push 781108903979.dkr.ecr.ap-south-1.amazonaws.com/node-api:latest

      artifacts:
        files:
          - '**/*'

  ** We can OVERRDIE buildspec.yml file without changing the Source code using -

    1. Using start-build api -
      
      aws codebuild start-build --project-name my-project --buildspecOverride buildspec-prod.yml

    2. In CodeBuild Project settings

    3. Using env variables
       build:
          commands:
            - npm run $BUILD_COMMAND
  
  * CodeBuild supports build caching using Amazon S3 and local storage but only useful if a subsequent 
    build lands on the same host (which is not guaranteed)

7. CodeBuild Agent -

  * Developers push bad code to main branch which result in Build failures
    To avoid this we can do two things -

    1. Build locally before merging (Most efficient, but requires CodeBuild agent to run locally on 
       developer infrastructure)

    2. After push Build feature branch before merging to main (More costly since feature branch build can
       be triggered multiple times)
  
  * To build locally we need to have CodeBuild Agent, which is a Docker based agent which runs the 
    buildspec.yml file locally, and if build is successful then only the developer merges the code

8. AWS CodeDeploy & its Hooks -

  * It is a deployment service that automates the application deployment to -
    EC2 instances / On-Premises servers
    ECS
    Lambda
    ASG

  * For EC2/OnPremise servers CodeDeploy Agent needs to be installed on the system
    CodeDeploy Agent has built-in revision cleanup using the :max_revisions: configuration
    unlike Jenkins, we don't have to write custom cleanup scripts for deployment artifacts

  * appspec.yml contains the configurations commands for deployment

  * It also supports Rollbacks for failed deployments, we can also trigger CloudWatch Alarms based on
    deployment status or metrics

    Rollback is done to the last stable version

  * Code Deploy Compute Platforms -

    | Platform        | Agent Required   | What Gets Deployed   |
    | --------------- | --------------   | -------------------- |
    | EC2/On-Premises | ✅ Yes          | Files/ZIP            |
    | ECS             | ❌ No           | New Task Definition  |
    | Lambda          | ❌ No           | New Function Version |

  * Each Compute Platform has different set of hooks -

    * For EC2/On-Premises -

      ApplicationStop
      DownloadBundle (No scripts allowed here, internal process i.e we cannot attach script here)
      BeforeInstall
      Install (No scripts allowed here, internal process i.e we cannot attach script here)
      AfterInstall
      ApplicationStart
      ValidateService

    * For Lambda (V1 to V2 version) -

      BeforeAllowTraffic
      AfterAllowTraffic

    * For ECS (Old Task Set (Blue) + New Task Set (Green)) -

      BeforeInstall
      AfterInstall
      AfterAllowTestTraffic
      BeforeAllowTraffic
      AfterAllowTraffic

    ** For Lambda/ECS hooks are Lambda functions whereas for EC2/OnPremise these are scripts

  * CodeDeploy supports resources from Amazon S3, GitHub, CodeCommit, CodePipeline Artifact and build 
    created locally CANNOT be referred without uploading it to S3

  ** CodeDeploy needs CodeDeploy service role/ IAM Role to call Lambda and not Lambda resource Policy

  * CodeDeploy doesn't works with Elastic Beanstalk since it itself contains deployment capabilities i.e
    it is itself a deployment orchestration service

9. CodeDeploy Groups vs CodeDeploy Application -
  
  * CodeDeploy Application - Logical container or Project eg - e-commerce, payment-service etc

  * Deployment groups = Deployment Group = WHERE + HOW to deploy
    i.e which service, deployment strategies, rollback etc
    
    Each group can deploy the same application with different configurations or revisions completely
    Example e-commerce, we can have dev-group, test-group and prod-group

10. AWS CodeArtifact -

  * It is a fully managed and scalable AWS service for package management (similar to NPN, MAVEN, PIP)
    where we can publish our packages and securely use them
  
  * To access CodeArtifact packages, IAM User must have valid policies attached so that it can access them

  ** For Cross Account access, we need to define Resource policy for CodeArtifact Domain, which allows
      another account access
  
  * Resource policies are attached Domain, inside Domain we have repositories/packages

11. AWS CodeGuru -

  * An ML-powered service for automated code reviews and application performance recommendations

  * It has Reviewer - who reviews the code for static code analysis 
    Profiler - which provides recommendations about application performance during runtime
  
  * MaxStackDepth – the maximum depth of the stacks in the code that is represented in the profile
    Example: if CodeGuru Profiler finds a method A, which calls method B, which calls method C, which 
    calls method D, then the depth is 4
    If the MaxStackDepth is set to 2, then the profiler evaluates A and B

12. AWS CodeStar -

  * It creates an entire CI/CD setup instead of manually creating each service
    It helps you assist in managing all facets of development and deployment
    It gives overview of all the stages at a single place
    
  * Legacy service, not used much
