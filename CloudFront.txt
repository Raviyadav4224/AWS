1. What is CloudFront - 

    * AWS CloudFront is a Content Delivery Network (CDN) which is used to deliver the content to the 
      Users with Low Latency and High Transfer Speed Using EDGE locations.
    
    * Instead of User calling the Origin directly, CloudFront caches the content closer to the user

    * User - Edge location - Origin(EC2, S3, ALB, API Gateway)

    * Example - User requests image.jpg
        Edge location checks
        Cache HIT - return immediately
        Cache MISS - request origin - store - return
    
2. Key Components -

    1. Edge location - 
        
        Data centers that caches the content closest to the User, Example Delhi, Mumbai, Singapore,
        Tokyo

    2. Origin -     

        The backend server where the content lives
        Supported Origins - EC2, S3, ALB/ELB, API Gateway, Media Servers

    3. Distribution -

        * It refers to the CloudFront configuration that defines
          Origin
          Caching Behavior
          Routing
          Security
        
        * It is of Two types 

            1. Web Distribution - For Http/Https based contents
            2. RTMP Distribution - For Media Streaming ( legacy , AWS Suggests to use HTTP Streaming)

3. Origin Access Control -

    * It prevents direct access to S3

    * Without protection User with S3 url can bypass the CloudFront
      User - S3 URL (Bypasses the CloudFront)

    * With OAC, S3 bucket policy allows only CloudFront
      User - CloudFront - S3

4. Signed URLs and Signed Cookies -

    * Signed URL is used to acces a single file

    * Signed Cookies are used to access multiple files
      Example - User Login - Cookie set - Access many videos (Netflix like streaming)

5. Cache Key -

    * CloudFront caches the objects based on Cache Key

    * Cache Key uniquely identifies an object in the cache

    * Cache Key = URL +  Some Additional Parameters

    * Based on Cache Key, CloudFront decides what to cache

    * Allowed values in a Cache Key -
        URL /images/logo.png
        Headers Accept-Language
        Cookies sessionId
        Query Strings ?id=123

    * Cache Control Headers -

        Origin can cache the content based on HTTP Headers
        Common Headers - Cache-Control, ETag, Last Modified Date, Expires
        Example - Cache-Control: public, max-age=3600 - caches the content for 1 hour

6. Cache Policy -

    * It defines what goes into the cache + TTL settings

    *   | Setting       | Purpose             |
        | ------------- | ------------------- |
        | Query strings | include/exclude     |
        | Headers       | include/exclude     |
        | Cookies       | include/exclude     |
        | TTL           | min / default / max |
    
7. Origin Policy -

    * Cache policy controls what goes into the cache while Origin policy decides what gets forwarded 
      to Origin
    
    * Example 
        
        User request:
            GET /product?id=10
            Header: Authorization
            Cookie: session

        Cache key uses: id

        But origin still needs:
            Authorization header
            session cookie

        So configuration:

            Cache Policy:

                Include query: id
                Exclude cookies
                Exclude headers

            Origin Request Policy:

                Forward Authorization header
                Forward cookies

8. Geo restriction -

    * CloudFront can restrict the content based on country and it can add it in whitelist or blacklist
      Example - Allow India , Block Russia, China

9. Cache TTL and Invalidation -

    * Cache TTL -

        | TTL         | Meaning                               |
        | ----------- | ------------------------------------- |
        | Minimum TTL | minimum caching time                  |
        | Default TTL | used if origin doesn't define caching |
        | Maximum TTL | maximum caching allowed               |

    * Cache Invalidation - 

        * Used when cached content needs to be removed before TTL expires
        * First 1000 invalidation paths for a MONTH are free beyond that are charged

10. CloudFront Behaviors -

    * It refers to how CloudFront handles request for specific paths 

    * Each behavior contains -
        Path
        Cache Policy
        Origin
        Origin Policy
        Viewer Request Policy
        Allowed HTTP Headers

11. Lambda@Edge -

    * It basically allows running Lambda functions at Edge locations

    * Execution Stage -

        | Event           | When                 |
        | --------------- | -------------------- |
        | Viewer Request  | before cache lookup  |
        | Origin Request  | before origin        |
        | Origin Response | after origin         |
        | Viewer Response | before response sent |

12. CloudFront Functions -  

    * It allows you to run lightweight(2MB Memory) functions at Edge locations with ultra fast latency

    * Specifically for Javascript Runtime Environments

    * It triggers on 

        Viewer Request
        Viewer Response
    
13. CloudFront + Web Application Firewall (WAF) -

    * It protects the CloudFront from attacks

    * Protect against SQL Injection, XSS, Rate limitinge etc

14. CloudFront Logs -

    * These are of two types -

        1. Standard Logs -

            Stored in S3
            Contains IP, Path, User Agent, Status, Edge location

        2. Real Time Logs -
            
            Streams logs immediately
            Used for security, monitoring, analytics

15. CloudFront Price Class - 

    * Price classes allow you to balance cost and performance by limiting which edge location your 
      distribution uses

    * Price Class All (Default) - It includes all Edge locations for best use

    * Price Class 200 - Covers most regions EXCLUDING the most expensive ones (typically South America 
      and Australia)

    * Price Class 100 - Covers the most EFFECTIVE regions (USA, Canada, Europe)

16. Origin failover -

    * CloudFront allows the User to configure Primary and Secondary Origins and also specify the HTTP
      error codes to trigger failovers
    
    * Example - 
        Primary Origin → S3
        Secondary Origin → Backup S3
        If Primary origin fails CloudFront automatically swithces to secondary origin
    
    * It is used for High Availability

17. Miscellaneous -

    * CloudFront is a Global service managed from us-east-1 (North Virginia)
    
    * CloudFront Distribution i.e its configuration is done in one region but that configuration 
      applies globally

    * CloudFront is Global and not region specific but the origin behind it depends
      If S3 - regional
        ALB - regional
        API Gateway - regional
        EC2 - deployed in AZ within region
    
    * For CloudFront + HTTPS - SSL certificate must be in us-east-1

    * S3 origin vs website endpoint - S3 origin supports OAC while website endpoint doesn't, 
      bucket must be public

    ** Origin Response Headers vs Viewer Request Headers -

        Origin Response Headers -

            These headers control how long CloudFront caches objects.
            These Headers are from Origin i.e S3, API or ALB
            Example response from origin: Cache-Control: max-age=300
            CloudFront caches the object for 300 seconds
            If origin is Amazon S3, this header is usually set as object metadata.
            Example metadata:
            Cache-Control: max-age=300

        Viewer Request Headers -

            Viewer headers are sent by the client (browser, Postman, curl).
            Example: GET /image.png, Cache-Control: max-age=30
            Viewer Cache-Control headers do NOT control CloudFront caching TTL.
            CloudFront ignores them for caching decisions. Clients should not control CDN cache behavior.
            *Viewer headers affect caching only if included in the cache key.
            Example cache policy includes: Accept-Language, User-Agent
            Then requests like: Accept-Language: en, Accept-Language: fr
