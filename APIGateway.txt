1. What is API Gateway -

  * It is an fully managed AWS service which acts as a single entry point for all the API's

  * It lets you -
    Create API's
    Expose backend services securely
    Route requests
    Rate limiting
    Throttle traffic
    Request/Response transformation
    Authenticate users
    Monitor API's

2. REST vs HTTP API -

  * REST provides more granular control, advance request/response transformation and security compared
    to HTTP
  
  * HTTP is cheaper, faster and simpler compared to REST

  * HTTP is mainly for simple API's and REST for entriprises API's

3. API Gateway Integration Types -

  * It means Where API Gateway sends the request, it can be a Lambda, any AWS service like DynamoDB,
    SQS or it can be custom http server
  
  * MOCK Integration -

    * No actual server and API Gateway returns a dummy response
      Mainly used for testing, Health checks, Dummy API's etc
  
  * HTTP Integration -

    * API Gateway forwards request to some HTTP endpoint, it may be any backend server, EC2, ECS etc
      API Gateway may transform the request
      Example - Client  GET /users/1, API Gateway may transform GET https://backend.internal/api/v2/users?id=1

  * HTTP_PROXY Integration -

    * Here request is forwarded as it is i.e No changes are done

    * Headers, Request body, response body are mostly preserved
  
  * Lambda Integration (Non-Proxy) -

    * API Gateway invokes Lambda.
      But API Gateway controls request format and response format

    * Example:

      Client request:

          {
          "username": "ravi"
          }

      API Gateway transforms into:

          {
          "user_name": "ravi",
          "source": "mobile-app"
          }

      Lambda receives transformed payload.
      Similarly response can also be transformed.

  * Lambda Proxy Integration -

    * API Gateway forwards entire HTTP request to Lambda
      Lambda handles everything.
    
    * Here API Gateway mostly acts as pass through
  
  * AWS Service Integration -

    * API Gateway directly talks to AWS services, No Lambda needed.

    * Example:
      Put item in DynamoDB
      Send message to SQS
      Start Step Functions
      Publish SNS

4. Mapping Templates -

  * It defines How API Gateway changes requests/responses i.e Mapping templates can be used to
    modify request / responses

  * Client Request => Method Request => Integration Request => Backend
    Backend Response => Integration Response => Method Response => Client

  * Rename / Modify query string parameters 
  
  * Modify body content
  
  * Add headers
  
  * It uses Velocity Template Language (VTL): for loop, if etc...
  
  * Filter output results (remove unnecessary data)
  
  * Content-Type can be set to  application/json or application/xml

5. Authorization -

  * When request comes to an API Gateway it authenticates and authorizes it

  1. IAM Authorization -

    * It uses IAM credentials to call API Gateway

    * API Method Authorization Type - AWS_IAM
      Now caller must sign request using -
      Access Key
      Secret Key
      SigV4 signing (SigV4 = Authorization Header OR X-Amz-Signature Query String)

    * API Gateway checks - Is caller authenticated? Does IAM policy allow execute-api:Invoke ?
    
    * With IAM auth -
      Authentication = AWS credentials
      Authorization = IAM policy

  2. Lambda Authorizer -

    * API Gateway calls a Lambda to Run custom code to decide whether request is allowed or not

    * Lambda doesnot returns response body rather it returns IAM policy document

    * Types of Lambda Authorizer -
      
      TOKEN Authorizer - 

        Receives Authorization: Bearer xyz
        Lambda validates token

      REQUEST Authorizer -

        Receives: headers, query params, path params
        More flexible

    * We can cache the result of Authorizer to reduce Lambda invocations
  
  3. Cognito User Pools -

    * User logs in via Amazon Cognito
      Cognito returns - JWT token
      Client sends token to API Gateway.
      API Gateway validates JWT automatically.
      No Lambda needed.

    * Cognito authorizer works with Cognito User Pools and NOT Identity Pools

  * API Keys & Usage Plans -

    * They do not provide Real Authorization or Identification

    * API Gateway checks valid key? linked usage plan?
      If yes - Request allowed

    * Usage Plans defines throttling and quotas

6. Throttling and Caching in API Gateway -

  * Throttling -

    * It is done mainly to protect the backend servers from sudden spikes in traffic and getting overloaded

    * AWS API Gateway uses TOKEN BUCKET Algorithm for this
      It has - 
        Rate capacity (Requests per second)
        Burst capacity (Temporary spike capacity)     

    * Here every request consumes 1 token, and at given rate it refills, if the Bucket is empty the 
      requests gets throttled with 429 Error code (too many requests)
    
    * Throttling levels -

      * Account Level (10,000rps) -

        This is the top level AWS limit, applies accross all regions, api's, stages etc
        Say your account has a limit of 
        Rate => 10,000 rps
        Burst => 5,000 rps
        If the traffic exceeds this level, API Gateway requests throtlles

      * Stage Level -
        
        Applied per stage say /dev stage, /prod stage
        We can configure different limits for each environment

      * Method Level -

        Applies at specific HTTP method + resource
        Suppose GET /products we can define 5000rps 

      * Usage Plan Level -

        Throttle per customer i.e define a limit for customer

  * Caching API responses -

    * Caching reduces the No. of API calls to the backend server

    * Default TTL = 300 seconds, Max 3600 seconds

    * Caches are defined per stage

    * We can override cache settings per method

    * We can Encrypt the cache data for sensitive API

    * Cache capacity - 0.5GB to 237GB

    * Cache invalidation, we can invalidate the entire Stage cache or per method cache
      
    * Client can send header: Cache-Control: max-age=0 to invalidate the cache immediately ( but must
      have IAM authorization)

7. Stages, Logging, Tracing & Metrics in API Gateway -

  * Stages -

    Stages are named environments pointing to deployments
    Examples: dev, test, qa, prod
    A stage contains runtime configurations for the API
  
  * Logging & Tracing -

    * API Gateway logging is configured per stage using Amazon CloudWatch

    * Execution Logs (For debugging)
      Managed by API Gateway, Used for debugging
      It Contains: request processing, auth failures, mapping errors, integration failures

    * Access Logs (For Caller Identity)
      Custom request logs, Used mainly for API usage tracking
      It has User-defined format
      Includes: requestId, Caller IP, route, status code, latency

    * Tracing - we can enable tracing per stage to help track the request flow
  
  * Metrics -

    * Metrics are by stage, Possibility to enable detailed metrics
    * CacheHitCount & CacheMissCount: efficiency of the cache
    * Count: The total number API requests in a given period.
    * IntegrationLatency: The time between when API Gateway relays a request to the backend and when
      it receives a response from the backend
    * Latency: The time between when API Gateway receives a request from a client and when it returns
      a response to the client. The latency includes the integration latency and other API Gateway 
      overhead.
    * 4XXError (client-side) & 5XXError (server-side)
    * API Gateway requests time out after 29 second maximum

8. API Gateway CORS -

  * It must be enabled when we receive API calls from different domains

  * It is a Browser enforced feature, so that only allowed domains can request data from backend servers
    Otherwise any server could request from other server
  
  * Before the actual request, Browser makes a preflight OPTIONS request with 
    Origin: https://app.com
    Access-Control-Request-Method: POST
    Access-Control-Request-Headers: Authorization

    Then Backend/API Gateway Responds
    Access-Control-Allow-Origin: https://app.com
    Access-Control-Allow-Methods: POST
    Access-Control-Allow-Headers: Authorization

    Then browser proceeds with actual request

9. Miscellaneous -
  
  ** API Gateway doesn’t have anything called Expire API versions after some days say after 
     90days automatically expire API version

  * To create different versions of API Gateway the most effective way is to use Stages
    
    Each stage gets unique endpoint, unique configuration, independent deployment and allows us to manage
    different versions of API differently without affecting the exisiting ones

    Stage variables can be used to provide context to the Lambda functions without changing anything
    at code level
  
  * API Gateway supports OpenAPI/Swagger API definitions for creation of API's

  * API Resource Policy can be used to whitelist specific IP's from accessing the API Gateway
    Create Resource Policy and control it with condition 

    aws:SourceIp - Only requests coming from these IPs are allowed
    aws:SourceVpce - Only requests through this VPC endpoint are allowed
  
  * API Gateway supports WebSocket API's and provides real time bidirectional communication
    WebSocket APIs typically route messages based on a field in the message body
    
    Example client message:

      {
        "action": "sendMessage",
        "message": "Hello"
      }

      API Gateway extracts - $request.body.action and routes to the matching route

  * Mechanism that CANNOT be used for authenticating with the API Gateway - STS
    That can be used are - Lambda Authorizer, Cognito User Pools, IAM Roles and Policies

  * API Gateway Request Flow -

    Client => Method Request => Integration Request(Map request body) => Backend(Lambda/HTTP/AWS Service) => 
    Integration Response => Method Response => Client
  
  * | Action                      | Meaning                                                                      | Traffic              |
    | --------------------------- | ---------------------------------------------------------------------------- | -------------------- |
    | **Enable Canary Release**   | Start testing a new deployment with a small percentage of production traffic | e.g. 95% old, 5% new |
    | **Promote Canary Release**  | Make the tested deployment the primary production deployment                 | 100% new             |
    | **Rollback/Disable Canary** | Stop using the canary and send all traffic back to the stable deployment     | 100% old             |
