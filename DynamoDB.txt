1. What is DynamoDB -

  * It is a highly scalable, fully managed AWS service

  * It stores data in Key-Value pairs i.e it is a distributed key value store

  * It doesnot supports joins/aggregations like traditional SQL db or MongoDB 

2. Basics -

  * Tables - Like SQL Table

  * Item - Equivalent to Row i.e Row in a Table

  * Attributes - Keys in Object, which can differ per object since no fixed schema

  * Data Types -

    * Scalar Types – String, Number, Binary, Boolean, Null
    * Document Types – List(Array), Map(key-value)
    * Set Types – String Set, Number Set, Binary Set

  * Partition Key (Primary Key) -

    * For each item Partition key must be UNIQUE and diverse so that data can be distributed

    * DynamoDB internally HASHES the Partition key and then decide where Data is stored

    * Example - userId, empId etc

  * Partition Key + Sort Key (Composite Key) -

    * The combination must be UNIQUE for each item
      Example - userId + orderId
    
    * Items are grouped using the Partition Key

    * Items within the same partition are stored sorted by Sort key

3. Read/Write Capacity Modes -

    * Throughput - 
        
      * It is the Read/Write Capacity of DynamoDB i.e How many reads/writes per second the 
        table can process
        Example - 100 reads/sec, 50 writes/sec

      * Why Throughput exists ? Since everything has limit, like DynamoDB internal CPU, Network etc
        has fixed limits, So it must control how much data it can receive else any user can overload it

    * Provisioned Mode -

      * Here we decide the Capacity i.e say 50 reads/sec and 100 writes/sec
      
      * If the traffic exceeds Throtlling happens

      * Used mainly for stable workloads

    * OnDemand Mode -

      * We dont configure anything, AWS automatically scales reads/writes dynamically
        Unlimited WCU/RCU and No Throtlling
      
      * We pay for per request of Read/Write, 2.5 times more expensive compared to Provisioned

      * Use for unpredictable traffic, sudden spikes etc

    * Write Capacity Unit (WCU) -

      * It is defined as 1 Write per second for an item upto 1KB size, For items larger than 1KB
        more WCU are required

        Example - 

          Item size = 1KB, Need = 10 writes/sec, then required WCU = 10 WCU
          Item size = 4.5KB, Need = 6 writes/sec, then required WCU = 6 * 5/1 = 30 WCU

    * Read Capacity Units (RCU) -

      * It is defined as 1 Strongly Consistent Read or 2 Eventually Consistent Read per second for an 
        item of size upto 4KB
      
      * Example - 

          10 Strongly Consistent Reads per second, with item size 4 KB, We need 10 ∗ 4/4 = 10 RCU
          16 Eventually Consistent Reads per second, with item size 12 KB, we need (16/2) * (12/4) = 24RCU

    ** Partitions Internals -

      * Data is stored in partitions

      * DynamoDB uses Partition keys which further goes through Hashing algorithm to decide which partition
        to go
        Example - userId(101) - Hash(101) - Partition A, userId(102) - Hash(102) - Partition B
      
      * Rather than storing all the data in one storage, DynamoDB distributes the data in different storages
        known as Partitions
      
      * Since data is distributed in different partitions Read/Write can happen in parallel and hence DynamoDB
        scales massively
      
      * Partition keys are very important, because of this DynamoDB stores data in different partitions
        and bad designing of partition keys leads to overload of same partition with more traffic and 
        it leads to Hot Partition

      * If the table has 10,000 WCU/RCU and all traffic goes to same partition it can still throttle
        Since each partition has limits

        Read Capacity Units - 3000 RCU's
        Write Capacity Units - 1000 RCU's
        
    * Throtlling -

      * When available Capacity is less than the requested Capacity, DynamoDB rejects the request, it is
        known as Throtlling

      * When request throttles, we get ProvisionedThroughputExceededException 

    * Strongly Consistent Read vs Eventually Consistent Read -

      Eventually Consistent Read (Default) -

        If we read just after a write, it is possible we get some stale data because of replication
        It is faster and cheaper
        Use case - Social media likes, Products etc

      Strongly Consistent Read -

        You will always get the latest commited data
        It is slower compared to Eventually Consistent Read and consumes more RCU
        Use case - Banking, checking balance after transferring money needs latest data

    * Replication - 
      
      * By default DynamoDB replicates data across multiple AZs within a region.
      
      * For Cross region replication, we need to use Global tables, and this process in Async in nature
        so it is always Eventually Consistent Read

4. Reading/Writing in DynamoDB -

  * Writing -

    * PutItem (INSERT OR REPLACE) - 

      It replaces the older object or inserts the new one
    
    * UpdateItem (Udpate and NOT REPLACE) -

      * Updates the existing object

      * It supports Update expressions 

        | Operation | Meaning              |
        | --------- | -------------------- |
        | SET       | add/update attribute |
        | REMOVE    | remove attribute     |
        | ADD       | increment numbers    |
        | DELETE    | remove from set      |

      * It is ATOMIC in nature means that two concurrent requests wont interupt or corrupt each other

  * Reading -

    * GetItem -

      * Fetches one item using Primary key, Fastest Read Operation

      * If Primary and Sort key both exists, both need to be specified while getting the data

    * Query -

      * Fetches multiple related items using -
        Partition key value (required) (=)
        Sort key value (optional) (=, <, <=, >, >=, Between, Begins with)
      
      * Why Query is faster ? 
        
        It doesnot queries the whole table since DynamoDB knows the Partition using partition key
    
    * Scan -

      Scan the entire table and then filter out data (inefficient)

      It consumes more RCU's
    
  * Deleting -

    * DeleteItem -

      * Deletes an individual item
      * Ability to perform a conditional delete

    * DeleteTable -

      * Delete a whole table and all its items
      * Much quicker deletion than calling DeleteItem on all items

  * Batch Operations -

    * Allows us to reduce Number of API calls

    * BatchWriteItem

      * Supports multiple PutItem, multiple DeleteItem
      * Up to 25 items in single request.
      * UnprocessedItems for failed write operations 

    * BatchGetItem -

      * Return items from one or more tables
      * Up to 100 items, up to 16 MB of data
      * UnprocessedKeys for failed read operations

  * PartiQL -

    SQL like syntax for DynamoDB, under the hood it still uses DynamoDB rules

    Example - GetItem API, you can write - SELECT * FROM Users WHERE userId='101'

5. Conditional Operations -

  * Conditional Writes -

    * Used when data is being written in to the DB (PRE-WRITE)

    * attribute_exists(attribute) - Checks attribute exists or not, if not then doesnot proceeds

    * attribute_not_exists(attribute) - Inserts only when the attribute doesnot exists

    * attribute_type(age, 'N') - Checks attribute type, i.e here age must be Number

    * contains(tags, :tag) - Checks whether a substring exists within another string or not
      It is less efficient
      Not allowed in KeyConditionExpression

    * begins_with(tags, :tags) - Checks whether value STARTS with something.
      It is more efficient
      It is index-friendly
      It can be used with sort keys in KeyConditionExpression

  * KeyConditionExpression vs FilterExpression vs ConditionalExpression -

    KeyConditionExpression - 
    
      * It is PRE-READ, used for Quering operations using keys
      * It is the most efficient since it uses Primary and Secondary keys to locate the data

    FilterExpression -

      * Used with Query/Scans, to filter data after it is read

    ConditionalExpression -

      * It is PRE-WRITE, i.e it evaluated before writing in DB
      
    | Function             | KeyConditionExpression | FilterExpression | ConditionExpression |
    | -------------------- | ---------------------- | ---------------- | ------------------- |
    | begins_with          | ✅                      | ✅                | ✅                   |
    | contains             | ❌                      | ✅                | ✅                   |
    | attribute_exists     | ❌                      | ✅                | ✅                   |
    | attribute_not_exists | ❌                      | ✅                | ✅                   |
    | size                 | ❌                      | ✅                | ✅                   |
    | attribute_type       | ❌                      | ✅                | ✅                   |

  ** BatchWriteItem(supports put and delete) and ConditionExpression is NOT supported, for that we have
     to use TransactWriteItems, This supports Put, Update, Delete, ConditionCheck

6. Handson Examples -

  * GetItem -

    aws dynamodb get-item \
    --table-name Users \
    --key '{
      "userId": {"S": "emp1"}
    }'
    
  * Update Existing Items -
    
   aws dynamodb update-item \
    --table-name Users \
    --key '{
      "userId": {"S": "emp1"}
    }' \
    --update-expression "SET salary = :newSalary" \
    --condition-expression "salary = :oldSalary" \
    --expression-attribute-values '{
      ":newSalary": {"N": "7000"},
      ":oldSalary": {"N": "5000"}
    }'

  * Insert Items -

    aws dynamodb put-item \
    --table-name Users \
    --item '{
      "userId": {"S": "emp1"},
      "name": {"S": "Aman"},
      "salary": {"N": "5000"},
      "department": {"S": "Engineering"}
    }'

  * DeleteItems -

    aws dynamodb delete-item \
    --table-name Users \
    --key '{
      "userId": {"S": "u-delete"}
    }' \
    --condition-expression "status = :s" \
    --expression-attribute-values '{
      ":s": {"S": "INACTIVE"}
    }'

7. Local Secondary Index vs Global Secondary Index -

  * Why these indexes are required ?
    
    DynamoDB queries the data faster using indexes like Primary and Secondary
    Now in order to query different data based on different attributes say base table PK userid, 
    SK orderId, To get all users with status PENDING - total scan would be required without using index, 
    so LSI and GSI exits to solve this problem
    
  * Local Secondary Index -

    Same partition key as base table
    Different sort key allowed
    Created only during table creation
    Supports strong consistency
    Shares partitioning with base table
    Good for alternate sorting/querying within same partition
    Cannot change partition access pattern
    10GB partition limit applies
    Usecase - Get all orders of USER U1 by status (since partition is same)

  * Global Secondary Index -

    Different partition key allowed
    Different sort key allowed
    Can be added later
    Separate storage and partitioning
    Separate scaling and throughput
    Enables completely new access patterns
    Most commonly used DynamoDB index
    Usecase - show ALL PENDING orders (old PK SK wont work and full table scan would be required)
